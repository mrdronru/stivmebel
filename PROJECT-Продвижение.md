**Статус: только обсуждается, не начато.** Зафиксировано здесь, чтобы не потерять идею
и детали к моменту, когда решим приступить.

### Идея

PHP-страницы сейчас рендерятся заново при каждом обращении к сайту — это лишняя
нагрузка там, где контент не меняется от запроса к запросу. Предлагается по
аналогии с Азбукой (`build.php` уже генерирует `abc/\*.html` из `.md` через
`header.php`/`footer.php`) расширить этот подход на весь сайт: превратить
контентные страницы в статический HTML, оставив PHP только там, где он
объективно нужен — серверная логика и секреты.

### Что остаётся на PHP всегда

* `send.php` — обработка формы, отправка в Telegram (токен бота, server-side по определению).
* `config.php`, `build.php`, `markdown-helpers.php` — служебные файлы, не отдаются напрямую.
* Любая логика, которая обязана выполняться на сервере при каждом запросе.

### Что переводится в статику

`index.php`, `about.php`, `price.php`, `catalog.php`, `video.php`, `favorites.php`,
`abcd.php` (сам список терминов, не только сгенерированные статьи), `politika.php`,
`soglasie.php` — весь контентный слой сайта.

### Решённые вопросы (на этапе обсуждения)

* **Переход одним рефакторингом**, не постранично — весь сайт за один заход.
* **URL без `.html`** — красивые адреса (`/about`, `/gallery`) через `.htaccess`,
не показывать расширение пользователю.
* **Пересборка после правок — нормальная практика**, уже привычна по работе
с Азбукой: после любого изменения `header.php`/`footer.php`/другой общей части
нужно заново запускать `build.php`, чтобы изменения попали на все страницы.
Это осознанно принятый компромисс, не недостаток подхода.

### Технические задачи, которые предстоят

1. **Маршрутизация в `.htaccess`.** ✅ Выполнено досрочно 11.07.2026 в рамках
SEO-доработок (см. раздел «SEO-доработки» ниже): чистые URL теперь канонические.
Для перехода на статику останется только поменять приоритет в правилах: если
существует `.html` — отдать его, иначе `.php` (на переходный период).
2. **Переписать все внутренние ссылки сайта.** ✅ Выполнено 11.07.2026: все
внутренние ссылки в `header.php`, `footer.php`, страницах, лендингах, `404.php`
и генераторе `build.php` переведены на чистые URL одним проходом.
3. **301-редирект со старых `.php`-адресов.** ✅ Выполнено 11.07.2026 —
см. раздел «SEO-доработки». Проиндексированные `.php`-адреса и `abc/*.html`
отдают 301 на чистые URL, SEO-вес переносится.
4. **`catalog.php` и `video.php` — главное изменение в рабочем процессе.**
Сейчас обе страницы читают список файлов через `glob()` при каждом заходе —
это и есть вся их «логика». В статике это означает: добавили фото/видео на
сервер → нужно зайти на `build.php` → страница пересоберётся. Раньше было
«просто залил файл — он сразу на сайте». Владелец предупреждён и согласен
на этот компромисс.
5. **`favorites.php`.** Список избранного хранится в `localStorage` браузера —
это уже клиентская логика, страница переносится в статику почти без проблем.
6. **`$canonical\_path` для каждой страницы.** Та же история, что уже решена для
Азбуки — при генерации из CLI `$\_SERVER\['REQUEST\_URI']` не определён, нужно
явно прописывать canonical/og:url для каждой генерируемой страницы.
7. **`build.php` должен научиться генерировать не только статьи Азбуки.**
Сейчас генератор заточен под `abc/\*.md → abc/\*.html`. Нужно расширить его
(или сделать параллельный генератор) для `index.html`, `about.html`,
`gallery.html` и т.д. — каждая страница сохраняет свою текущую бизнес-логику
(например, `glob()` для фото в галерее), просто результат уходит в файл,
а не в ответ на HTTP-запрос.

### Перед началом работы — план первого шага

Когда будем готовы перейти от обсуждения к реализации: начать с инвентаризации —
собрать полный список всех файлов, всех внутренних ссылок и всех мест, которые
нужно изменить, прежде чем писать код. Масштаб рефакторинга подразумевает
системный проход по всем PHP-страницам, общим частям (`header.php`/`footer.php`),
`.htaccess` и всем внутренним ссылкам разом — стоит исключить риск что-то упустить.

подробности, как в случае с Липатовым). Полностью редактируемый `.pptx`.

**Июль 2026: полностью пересобрана с нуля** по отдельному ТЗ (уровень
«европейского бренда»: минимум текста, максимум воздуха, референсы
Apple/Notion/Linear/Poliform/Boffi). Фото проектов и команды взяты из прежней
версии, тексты переработаны под новую структуру.

**Статус (уточнено 12.07.2026): в работе — прежняя чёрная
`Stiv_Interiors_dlya_dizaynerov_v2.pptx` (8 слайдов)**, именно она уходит
дизайнерам в follow-up. Новая белая версия — черновик: в ней плейсхолдеры
кейсов и маленькие фото команды (см. TODO ниже), отправлять её нельзя.
Переход на белую — после доработки, срок не назначен.

**Бренд и контакты в презентации:** «Стив Интерьеры», телефон
`+7 (969) 256 23 23`, Telegram `t.me/stiv_interiors`, почта
`alex@stivinteriors.ru`, сайт `stivmebel.ru` (изначально Claude поставил
`stivinteriors.ru`, исправлено на основной домен — `stivinteriors.ru` лишь
редиректит на `stivmebel.ru`).

**Как собрана и как пересобирать:** генератор `build.js` (Node.js,
библиотека `pptxgenjs`) — тот же подход, что `kp_build.js` и
`build_strategy_docx.js`: все тексты, координаты и цвета правятся в скрипте,
затем `node build.js`, файл пересоздаётся.

**Папка `assets/` — обязательна для пересборки**, `build.js` без неё не
работает (лежит рядом со скриптом, передана архивом
`presentation_assets.zip`). Состав и происхождение файлов:
* `cover.jpg` — вертикальное фото оливковой кухни (обложка); из старой
презентации, `ppt/media/image1.jpg`
* `case1.jpg` — кухня, серый мат + деревянный остров (кейс 01); `image13.png`
* `case2.jpg` — светлая кухня с деревянным островом (кейс 02); `image15.png`
* `case3.jpg` — детская, зелёная эмаль с фрезеровкой (кейс 03); `image12.png`
* `team_alexander.jpg`, `team_stanislav.jpg` — фото команды 240×240;
`image25.png` и `image24.png` соответственно (подлежат замене, см. TODO)
* `qr.png` — QR на `https://t.me/stiv_interiors`, сгенерирован питоновской
библиотекой `qrcode`; при смене ссылки перегенерировать

При замене фото достаточно положить новый файл под тем же именем — координаты
и кадрирование (`sizing: cover`) в `build.js` подгонят его сами.

**Дизайн-система презентации** (не совпадает с сайтом — это сознательно,
аудитория другая): белый фон, чёрный `#141414`, серый текст `#6E6B64`,
подложки карточек `#F6F5F2`, акцент тёмно-оливковый `#5A5F46`, шрифт Inter,
формат 16:9. Решение по фону принято осознанно: обсуждали тёмный вариант и
«сэндвич» (тёмные обложка и контакты, белая середина) — отклонено, белый
по ТЗ.

**Важно про шрифт Inter:** не входит в стандартный Office. На каждой машине,
где презентацию открывают или редактируют, Inter нужно установить
(бесплатный, rsms.me/inter) — иначе PowerPoint подставит другой шрифт и
вёрстка поползёт.

**Структура (10 слайдов):**
1. Обложка 40/60 — заголовок «Производственный партнёр для дизайнеров
интерьера», слоган «Реализуем проекты так, как они задуманы», одно большое
фото справа
2. Почему дизайнеры работают с нами — 4 карточки (договор, ваш процесс,
нестандартные решения, один ответственный)
3. Что можем реализовать — 4 колонки (корпуса / фасады / дополнительно /
производство)
4–6. Три кейса — одно большое фото + карточка проекта (объект, город, м²,
изготовили, материалы, сроки)
7. Как строится работа — 7 этапов на одной горизонтальной линии
8. Отзывы — 3 карточки, только текст
9. Команда — Александр (работа с дизайнерами), Станислав (производство)
10. Контакты + QR на Telegram

**Урок вёрстки pptx:** многострочные заголовки бить на строки явными
переносами и давать текстовому блоку высоту с запасом — PowerPoint переносит
текст иначе, чем предпросмотр LibreOffice, и при нехватке высоты блоки
наезжают друг на друга (первая версия обложки именно так и сломалась).

**Длинное тире:** в этой презентации присутствует (пришло из текста ТЗ) —
осознанное исключение из глобального правила проекта, Александр при
необходимости правит руками.

## Каталог работ 2026 — «Стив Интерьеры. Каталог работ 2026»

для дизайнеров** (раздел выше) — это разные документы для разных аудиторий:
здесь B2C-клиент, там дизайнер-партнёр. Используется для скачивания с сайта и

### Попытка пересборки в новом макете — НЕ УДАЛАСЬ

13–14.07.2026 каталог был пересобран с нуля программно (python-pptx). Причины
пересборки: старый макет признан хрупким (автоподборы + табы + чужой шрифт) и
чужеродным по стилю (тропический шаблон Google Slides с пальмами не имеет
отношения ни к мебели, ни к бренду).

**Что было сделано:** новый макет 16:9, 78 слайдов, кремовый фон, фирменный
коричневый, логотип SV, шрифт Arial (есть на всех машинах — проблема
подстановки шрифтов исключена). Единая сетка проектного слайда: **одна большая
hero-фотография слева на всю высоту слайда**, справа — рубрика · номер,
название, слоган, состав «метка — значение», две миниатюры. Добавлены слайд
«Почему нам доверяют», 7 разделителей по типам мебели, 5 слайдов отзывов с
выносными цитатами, финал с подписью «Станислав · Стив Интерьеры» и QR-кодом.
Вес — 9 МБ PPTX / 10 МБ PDF.

**Почему забраковано (решение Александра 14.07.2026):**
* **Hero-фото выбраны неправильно.** Скрипт брал героем самое крупное по
  площади фото со старого слайда — этот критерий не совпадает с «главным»
  фото проекта.
* **Вертикальные фото испорчены кадрированием.** Много исходных фото
  вертикальные; при вписывании в горизонтальную hero-рамку центр-кропом
  срезаются важные части — детали мебели и головы людей.

Вывод на будущее: **автоматический выбор и кадрирование фото под жёсткую
сетку не работает** на этом фотоархиве. Если возвращаться к идее нового
макета — hero для каждого проекта нужно выбирать вручную, а сетку делать
адаптивной к вертикальным фото (например, вертикальная hero-колонка уже, с
местом под текст сбоку, вместо кропа).
