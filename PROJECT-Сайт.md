# Стив Интерьеры — Сайт (программная часть)

Часть проекта «Стив Интерьеры». Общий слой (контакты, правила, репозиторий,
уроки работы с инструментами) и карта документа — в `PROJECT.md`.
Смежные части: `PROJECT-Продвижение.md`, `PROJECT-Продажи.md`.
Ссылки «см. раздел … ниже / в конце файла» могут вести в другой файл части.

---

## Стек

* **PHP** — шаблонизация через `include 'header.php'` / `include 'footer.php'`
* **Vanilla JS** — без фреймворков
* **CSS custom properties** — единая дизайн-система

**Правила дизайна — кнопки (зафиксировано 14.07.2026 после двух повторов
одной ошибки):** любая кнопка и CTA оформляются ТОЛЬКО сплошной заливкой
с контрастным текстом — никаких контурных/прозрачных кнопок и никогда цвет
кнопки не должен совпадать или сближаться с фоном блока. Канон: на светлом
фоне (cream/warm-white) — `.btn-primary` как есть: заливка `--ink`, текст
`--warm-white`; на тёмном фоне (`--ink`) — заливка `--gold`, текст `--ink`,
ховер `--gold-light` (пример: `.project-cta-btn` в projectstyle.css).
Стандартный `.btn-primary` на тёмном блоке ставить нельзя — чернила на
чернилах не читаются как кнопка. Перед сдачей любой страницы проверять
кнопку на каждом фоне, где она встречается.

**Правила дизайна — размеры и значения (дизайн-система, зафиксировано
17.07.2026).** Главная — эталон дизайн-системы. Все значения (размеры
шрифтов, прозрачности, трекинг, длительности переходов, цвета) берутся
только из токенов в `:root` `style.css` (полный список — в разделе
«CSS-переменные» ниже). Никакая страница (внутренние, лендинги, Азбука,
проекты) не должна вводить размер/цвет/значение, которого нет в системе.

*Как понимать относительные просьбы.* «Сделай крупнее / мельче / плотнее /
светлее / быстрее» и т.п. — это НЕ повод писать произвольное число. Это
значит взять **ближайшую подходящую ступень из уже существующей шкалы**:
соседний `--fs-*`, соседнюю альфу `--ink/ww/gold-XX`, соседний `--ls-*` или
`--dur-*`.

*Если подходящего токена в системе нет* — не заводить значение молча.
Сообщить об этом и вместе с Александром выбрать один из двух путей:
(1) добавить новую ступень в `:root` — если это осмысленное общее значение,
которое пригодится и дальше; либо (2) сделать **разовый литерал прямо в
страничном CSS второго уровня** (`aboutstyle`/`landing`/…) — если значение
нужно ровно один раз и в шкалу не просится (такие one-off в `:root` не
выносим). Новый токен в `:root` заводится только под реальный кластер
употреблений и только с одобрения Александра.
* Хостинг: **Reg.ru** shared. Для форм сайта (`send.php`): `fsockopen` к внешним SMTP заблокирован, `curl` к внешним хостам работает, `mail()` принимает письма, но они не доходят (нет DKIM на уровне PHP-отправки) — поэтому заявки с форм идут в Telegram Bot API, а не на почту. Это ограничение не относится к отдельному процессу массовой рассылки дизайнерам (Python `smtplib` через SSH — см. раздел «B2B-рассылка дизайнерам» в конце файла), там SMTP настроен и работает.

### config.php — единый источник важных данных

**Все чувствительные и часто меняемые настройки хранятся только в `config.php`.** Этот файл подключается через `require\_once \_\_DIR\_\_ . '/config.php'` в начале каждой страницы и в `send.php`.

Что хранится в config.php:

* `SITE\_PHONE`, `SITE\_PHONE\_HREF` — номер телефона (меняйте только здесь)
* `SITE\_EMAIL`, `SITE\_EMAIL\_HREF` — email
* `SOCIAL\_TG`, `SOCIAL\_VK`, `SOCIAL\_MAX` — ссылки на соцсети (иконки в футере и попап-форме)
* `SOCIAL\_VK\_PAGE`, `SOCIAL\_TG\_CHANNEL`, `YANDEX\_MAPS\_URL` — ссылки на страницы/каналы и карту (текстовые ссылки в футере и блоке «Контакты»)
* `SITE\_NAME`, `SITE\_TAGLINE` — название и слоган
* `YANDEX\_METRIKA\_ID` — ID счётчика Яндекс.Метрики
* `TG\_TOKEN`, `TG\_CHAT\_ID` — токен Telegram-бота и ID чата получателя заявок
* `PRICES` — цены «от» (₽/п.м.) по категориям: wardrobes (шкафы и шкафы-купе,
включая прихожие), kitchens (кухни), closets (гардеробные), kids
(детские/столы/стеллажи, `from => null` = «расчёт после замера»). Единый
источник для блока «Наши цены» на главной, таблицы ориентиров и meta
description на `price.php`. «Шкафы» на главной и «Шкафы и шкафы-купе» на
/price — одна категория `wardrobes` с одной ценой. Добавлено 12.07.2026;
тогда же категории приведены в порядок: бывшая строка «Прихожие, шкафы»
(от 44 900) влита в wardrobes, её цена стала ценой категории, прежняя цена
wardrobes 89 000 ₽ убрана с сайта полностью (решение Александра).
* `price\_from($key)` — функция форматирования: `price\_from('kitchens')` →
«от 59 500 ₽»; для `from => null` возвращает «расчёт после замера».

**Важно:** не хардкодить контакты, токены, ID или цены напрямую в страницах — всё через константы из config.php. Цены менять только в массиве `PRICES` — главная, страница цен и её description обновятся сами.

Медиафайлы:

* `images/` — фото для index (k01–k08.jpg, m01–m06.jpg) и price (p01–p03.jpg — примеры работ на `price.php`); плюс `hello-poster.jpg`, `stas01.png` (см. ниже)
* `testimonials/` — отзывы для блока на `about.php`: фото и видео вперемешку (`001.png`, `004.jpg`, `.mp4` — любой из jpg/jpeg/png/mp4), нумерация `NNN` в имени = порядок. Список читается автоматически через `glob()` — добавлять в код вручную не нужно
* `projects/` — фото и чертежи страниц раздела «Наши проекты» (формат `cc-NN.jpg`, `cc-before.jpg`, `cc-plan-N.png`; префикс = код проекта)
* `gallery/` — фото галереи, формат `NNNN-category` (`0001-kitchen.jpg` и далее; нумерация растёт по мере добавления, на 17.07.2026 — сплошная `0001`–`0177`, категории: kitchen, wardrobe, hallway, kids, other)
* `video/` — видео: `NNNN.mp4` + `NNNN.jpg` (обложка) + опционально `NNNN.md` (подпись).
Список читается автоматически через `glob()` в `video.php` — без `.md` видео
показывается без подписи. См. раздел «Видео» ниже.
* `hello.mp4` — видео Станислава: используется на `about.php` (в круге) и на `land-ng.php` (полноэкранное в блоке о Станиславе). На `land-cc.php` в этом блоке вместо видео — фото `stas01.png`

### CSS-переменные (style.css)

Обновлено 17.07.2026 — большая консолидация: разрозненные литералы по
страничным CSS сведены в единый набор токенов в `:root`. Правила

|Файл|Что содержит|

Адаптив — основной брейкпоинт: `@media (max-width: 900px)`. Дополнительно в `gallerystyle.css` и `favoritesstyle.css` есть `@media (max-width: 500px)` — осознанное исключение для переключения сетки фото с 2 колонок на 1 на очень узких экранах, а не случайный третий брейкпоинт.

### Цепочки подключения и риск переопределений

Файл позже в цепочке побеждает при равной специфичности селектора — это
нормальный каскад CSS, но именно на стыке файлов рождаются «тихие» баги:
страничный файл переопределяет правило из общего `style.css`, и
переопределение либо теряет смысл исходного значения, либо расходится по
брейкпоинтам с третьим файлом в цепочке.

|Страница|Цепочка|
|-|-|
|index.php|style.css|
|about.php|style.css → aboutstyle.css|
|catalog.php|style.css → gallerystyle.css|
|favorites.php|style.css → gallerystyle.css → favoritesstyle.css|
|video.php|style.css → videostyle.css|
|price.php|style.css → pricestyle.css|
|abcd.php / abc/\*.html|style.css → abcstyle.css|
|land-ng.php|style.css → landing.css|
|land-cc.php|style.css → landing.css|
|land-mos.php|style.css → landing.css|

`favorites.php` — единственная страница с цепочкой длиной три файла, то есть
с наибольшим риском такого конфликта. Реальный случай: `favoritesstyle.css`
на мобильном переопределял `.fav-header { padding: 3rem ...}`, обнуляя
`padding-top: 7rem`, который `.gallery-header` в `gallerystyle.css` намеренно
ставил под высоту `position: fixed` навигации (`nav` в `style.css`) — в
результате шапка перекрывала заголовок страницы. Это и есть целевой класс
багов для процедуры проверки ниже (см. чеклист «CSS — консистентность»).

### Отзывы на лендингах — testimonials-grid (добавлено 16.07.2026)

`aboutstyle.css` использует ленту `.testimonials-strip` с горизонтальным
`overflow-x: auto` — на лендингах это на практике давало проблему: из 8
карточек было видно только 4–5, дальше нужно было скроллить, что не всегда
очевидно. В `landing.css` добавлены два новых класса-варианта:

* `.testimonials-grid` — адаптивная CSS-сетка (4 колонки → 3 → 2 на
  мобильных) вместо скролла, показывает сразу все карточки.
* `.testimonials-grid--quotes` + `.tq-card` (`.testimonial-quote`,
  `.testimonial-author`) — карточка «фото + текст цитаты + автор», для
  случаев, когда у отзыва есть реальный текст, а не только фото/видео.
  Используется на land-mos.

## Формы — ID всех элементов

Все формы используют `getElementById`, никаких `querySelector` для элементов форм.

**Honeypot (антиспам, добавлено 17.07.2026).** В каждой форме есть скрытое поле
`name="website"` с классом `.hp-field` (уводится за экран в `style.css`,
`tabindex="-1"`, `autocomplete="off"`, `aria-hidden`). ID полей: `fwebsite`
(index), `pfwebsite` (popup/footer), `mWebsite` (catalog и favorites),
`quizWebsite` (квиз — общий `quiz.php`: land-ng + land-mos). JS каждой формы читает это поле и кладёт в тело
запроса как `website`. Человек поле не видит и оставляет пустым; бот-автозаполнитель
его заполняет. `send.php` при непустом `website` тихо возвращает `ok:true` и
заявку НЕ отправляет (чтобы не подсказывать боту о ловушке). При добавлении новой
формы — не забыть про honeypot-поле и передачу `website` в `send.php`.

## Яндекс.Метрика — цели

Хелпер `ymGoal(name)` определён в `footer.php` и доступен глобально.
Все вызовы через `ymGoal(...)`, не напрямую через `ym(...)`.

|Идентификатор|Событие|Где срабатывает|

**Цели в коде vs. настройки счётчика (найдено 18.07.2026).** `ymGoal(...)`
(→ `reachGoal`) отправляет событие всегда, но в отчётах Метрики оно считается
только если в счётчике заведена цель типа «JavaScript-событие» с
идентификатором, **точно совпадающим** с тем, что шлёт код. При проверке квиза
на land-mos нашли рассинхрон: в счётчике была лишняя цель `quiz_step_3`

не было `quiz_step_6` (код её шлёт, но событие уходило в никуда). Исправлено на
стороне Метрики: у существующей цели `quiz_step_3` поменяли **идентификатор**
(не только подпись — считается по идентификатору) на `quiz_step_6`.

Актуальный набор целей в счётчике 108718941 после правки: `quiz_open`,
`quiz_step_4`, `quiz_step_5`, `quiz_step_6`, `quiz_complete` (+ `click_phone`,
`form_submit` и т.д.). Воронка квиза: открыт → шаг 4 → шаг 5 → шаг 6 →
завершён. Шаги 1–3 и экран контактов (шаг 7) своих целей не имеют — если
понадобится мерить отвал на форме, добавить цель на показ шага 7 в `quiz.php`.

## Квиз — общий файл quiz.php (рефакторинг 16.07.2026)

Раньше у каждого лендинга была своя копия квиза прямо в разметке страницы —
land-ng и land-mos успели разойтись (разное число шагов, разные вопросы,
разные способы связи), и точно так же разошлись бы любые следующие
лендинги. Теперь квиз — один файл `quiz.php`, подключается через `include`
там, где нужен:

```php
<?php $quiz_source = 'quiz-ng'; include 'quiz.php'; ?>
```

`$quiz_source` — обязательная переменная, задаётся ПЕРЕД `include`, уходит в
`send.php` как `source` (например `quiz-ng`, `quiz-mos`) — по ней видно, с
какого именно лендинга пришла заявка, и в Telegram, и в Метрике. Если не
задать — используется `'quiz'` (не ломается, но обезличено).

Сейчас подключён на `land-ng.php` (`source: 'quiz-ng'`) и `land-mos.php`
(`source: 'quiz-mos'`). На `land-cc.php` квиза по-прежнему нет.

**Важно:** `quiz.php` — единственный источник правды для вопросов и логики
квиза. Менять шаги/вопросы/способы связи нужно только там — правка сразу
применяется на всех лендингах, которые его подключают. Не копировать
разметку квиза обратно в конкретный `land-*.php`.

### Шаги (7 содержательных + экран входа + экран «спасибо»)

0. Экран входа — кнопка «Начать» (`startQuiz()`)
1. Что хотите сделать? — Кухню / Шкаф / Гардеробную / Несколько предметов мебели
2. Для какого помещения нужна мебель? — **пропускается автоматически**, если
   на шаге 1 выбрали «Кухню» (ответ подставляется как «Кухня», переход сразу
   на шаг 3; кнопка «Назад» с шага 3 в этом случае ведёт на шаг 1, а не на
   шаг 2 — флаг `roomSkipped` в JS)
3. Какой примерно размер помещения?
4. Какая задача сейчас самая важная?
5. На какой ориентир по стоимости рассчитываете? (5-й вариант — «Пока
   сложно оценить»)
6. Когда планируете заказать мебель?
7. Контакт — имя + способ связи: Телефон, WhatsApp, Telegram, Макс, Email

Экран «спасибо» — `data-step="8"`.

### send.php и source

`send.php` определяет заявку с квиза по префиксу: `strpos($source, 'quiz')
=== 0` — значит `quiz`, `quiz-ng`, `quiz-mos` и любой будущий
`quiz-<новый-лендинг>` обрабатываются одинаково (пропуск проверки
«похоже на телефон» для нетелефонных способов связи вроде Telegram/Email).
Подписи в Telegram берутся из массива `$quizSourceLabels` в `send.php` — при
подключении квиза на новый лендинг туда стоит добавить строку
(`'quiz-<название>' => 'Квиз (...)'`), иначе в Telegram будет просто «Квиз»
без уточнения, с какой страницы.

**Найденный и исправленный баг:** до этого рефакторинга сравнение было
`$source === 'quiz'` дословно — `land-mos.php` уже отправлял `'quiz-mos'`,
из-за чего для Telegram/Email в квизе на land-mos срабатывала проверка
«номер телефона обязателен», и заявка тихо отклонялась (без видимой ошибки
у пользователя, просто кнопка «Попробовать ещё раз»). Исправлено вместе с
переносом квиза в общий файл.

## send.php — отправка заявок

**Telegram Bot API через curl.**

```
Bot token: см. BotFather
Chat ID:   см. BotFather (@mrdronru — Alexandr Dranitskiy, владелец)
```

Принимает JSON: `name`, `phone`, `type`, `comment`, `photos\[]`, `source`.
Возвращает `{"ok":true}` или `{"ok":false,"error":"..."}`.

Поле `source`: `main` (форма index), `popup` (попап), `gallery` (модал галереи), `project-cc` (попап со страницы проекта; в Telegram — «Страница проекта (Сердце Столицы)»). Механика: попап в footer.php берёт источник из `window.popupSource || 'popup'` — страница проекта задаёт `window.popupSource='project-cc'` перед подключением футера; новые страницы раздела делают так же со своим кодом.

**Квиз — источники вида `quiz-<лендинг>`.** С 16.07.2026 квиз общий (`quiz.php`, см. одноимённый раздел), и `source` приходит не просто `quiz`, а с суффиксом лендинга — `quiz-ng`, `quiz-mos` и т.д. `send.php` матчит такие значения по префиксу (`strpos($source, 'quiz') === 0`), а не точным сравнением — это и определяет, пропускать ли проверку «похоже на телефон» для нетелефонных способов связи (Telegram/Email/и т.п.). Подписи для Telegram-сообщения берутся из массива `$quizSourceLabels`; при подключении квиза на новый лендинг нужно добавить туда строку `'quiz-<название>' => 'Квиз (...)'`, иначе в сообщении будет просто «Квиз» без уточнения страницы.

**`parse\_mode` не используется — сообщения отправляются как обычный текст.** Раньше использовался `Markdown` с неполным экранированием (`tg\_escape`), из-за чего спецсимволы в именах/телефонах/комментариях могли ломать запрос к Telegram API. Теперь `parse\_mode` убран из вызова API целиком, разметка `\*жирный\*` не используется — пользовательский ввод безопасен в любом виде.

### Картинки в статьях (добавлено 15.07.2026)

До статьи `ergonomika-kuhni` (первая статья с картинками, letter «Э») в
Азбуке не было ни одной иллюстрированной статьи — парсер и структура папок
были рассчитаны только на текст. Конвенция:

* **Хранение:** `abc/images/<slug>/` — своя подпапка на каждую статью с
  картинками, по её уникальному slug. Не пересекается с `gallery/`,
  `images/`, `projects/`, `testimonials/`, `video/`. В `.gitignore` строка
  `abc/images/` — картинки статей, как и остальные медиапапки, не хранятся
  в git, заливаются на сервер напрямую.
* **Markdown:** `markdown-helpers.php` → `inlineMarkdown()` поддерживает
  `![alt](путь)`. Regex картинки стоит ОБЯЗАТЕЛЬНО до regex ссылки
  `[текст](ссылка)` — у них общий "хвост" `(...)`, при обратном порядке
  ссылка перехватывает картинку раньше и превращает её в битую ссылку.
  Путь в `![]()` указывается относительно `abc/`, например
  `images/ergonomika-kuhni/01-vytyazhka.jpg` — резолвится корректно от
  чистого URL `/abc/<slug>` (та же логика, что у ссылок «пред/след»).
* **Стили:** `abcstyle.css` → `.abc-article-body img` (картинка рендерится
  парсером как `<p><img></p>`, стиль без обёртки).

**Паттерн «иллюстрация + текст-панель»** — если источник картинки сырая
инфографика с текстом внутри самой картинки: НЕ кропить текстовый блок как
есть — шрифт и отступы исходника почти никогда не совпадают с сеткой кропа
(текст обрезается с одного края или остаётся лишний воздух с другого).
Вместо этого кропится только иллюстрация, а короткий текст рядом рендерится
заново (Pillow, `ImageDraw` + `ImageFont`, шрифт `LiberationSans-Bold.ttf`,
фон `--cream` `#F5F2EC`, золотая черта-акцент `--gold` `#B8975A` сверху,
авто-подбор размера шрифта под длину текста). Даёт предсказуемый результат
без ручной подгонки кропа под каждый текст.

### Апскейл иллюстраций — рецепт для песочницы (добавлено 15.07.2026)

Песочница Claude стартует с нуля в каждой сессии — если апскейл понадобится
снова, вот рабочий рецепт (проверено на статье `ergonomika-kuhni`):

* Пакет `realesrgan-ncnn-py` с PyPI — модели уже внутри wheel (~45 МБ), не
  нужно тянуть веса отдельно (GitHub releases/raw.githubusercontent.com/
  huggingface.co в песочнице заблокированы allowlist'ом сети).
* Не хватает системных библиотек `libvulkan.so.1` и `libomp.so.5` — root
  недоступен, `apt-get install` не работает. Решение: скачать `.deb` пакеты
  напрямую с `archive.ubuntu.com` (доступен без ограничений) —
  `vulkan-loader` (`libvulkan1_*.deb`) и `llvm-toolchain-14`
  (`libomp5-14_*.deb`), распаковать через `dpkg-deb -x <файл> <папка>` (не
  требует root), путь до `.so`-файлов добавить в `LD_LIBRARY_PATH`.
* `Realesrgan(gpuid=-1, model=3)` — `gpuid=-1` включает CPU-режим (в
  песочнице нет GPU/Vulkan-устройства, но сам загрузчик `libvulkan.so.1`
  всё равно должен физически присутствовать, иначе `.so`-обёртка не
  импортируется). `model=3` — `realesrgan-x4plus-anime`, лучше подходит для
  плоской line-art графики (чистые контуры, текст на иллюстрации), чем
  фото-модель `model=4` (`realesrgan-x4plus`).
* ~7–15 сек на одну картинку на CPU.

## Лендинг land-mos.php — архитектура и решения

Третий лендинг, для широкой аудитории Москвы (группа 3 в кампании №4, копия
настроек группы 2, геотаргетинг — вся Москва). Делит общий `landing.css` и
шапку `.ln-nav` с land-ng/land-cc, но своя аудитория и тональность: люди,
которые понимают, что «хорошо и персонально» стоит дороже, чем «как
получится», но не ищут слово «премиум» — без «премиум/элит/роскошь» в
текстах, конкретика вместо образов (материалы, гарантии, сроки).

### Кнопка «Написать нам» вместо WhatsApp — общий дизайн-принцип

В hero и в финальном CTA кнопка ведёт не на `wa.me`-ссылку, а открывает
попап-форму (`openContactPopup()`). Причина — в России WhatsApp и Telegram
работают только через VPN, это неудобно и есть не у всех: прямая
CTA-кнопка на мессенджер отсекает часть аудитории уже на клике. Актуально не
только для этого лендинга — при создании новых страниц с CTA на мессенджер
по умолчанию стоит предлагать форму, а не прямую ссылку на чат.

### Цены — динамически из config.php

Текст про цены не хардкодит цифры, а вызывает `price_from('kitchens')` /
`price_from('wardrobes')` / `price_from('closets')` — все три цены (кухни,
шкафы, гардеробные) берутся из `PRICES` и показываются единообразно «от N ₽
за п.м.». При изменении цен в `PRICES` текст на странице обновится сам, без
правки land-mos.php. (Кухня раньше была захардкожена «от 300 000 ₽» — заменено
17.07.2026 на `price_from('kitchens')`, за п.м., как остальные.)

`price_from()` вставляет внутрь цены неразрывные пробелы U+00A0 (и в
разделитель тысяч, и перед ₽), чтобы «от 59 500 ₽» переносилось только
целиком — ни «59 500», ни знак ₽ не съезжают на другую строку. Используется
реальный символ `"\xC2\xA0"`, а НЕ entity `&nbsp;`: строка идёт и в
meta-description (`price.php`), где entity экранировалась бы и показалась
буквально. Правка глобальная — действует на всех страницах, где выводится
цена (land-mos, главная, страница цен). Добавлено 17.07.2026.

### Портфолио — истории проектов (переделано 17.07.2026)

Те же 5 фото: `gallery/0002-kitchen.jpg`, `0073-hallway.jpg`,
`0091-kitchen.jpg`, `0109-kitchen.jpg`, `0056-wardrobe.jpg`. Но вместо
плотной фото-мозаики с однострочными подписями (как на land-ng/land-cc) —
**просторные карточки**: фото + заголовок + короткая история проекта в
2–3 предложения по схеме «задача → решение → почему именно так». Это
операционализирует сквозную идею лендинга (см. ниже «мы объясняем
решения»): портфолио не показывает, а рассказывает.

Карточный вид сделан модификатором `.l-portfolio--stories` на секции
работ (НЕ на секции отзывов, у неё тот же класс `.l-portfolio`) —
scoped-переопределения в `landing.css` не трогают общую мозаику
`.l-pf-*`, которую делят land-ng и land-cc (проверено: у них 5 работ
остались мозаикой). Текст истории — класс `.l-pf-story`. 5-й проект
переименован с «встроенный шкаф-купе под нишу» на «Мебель для комнаты:
удобства, которые замечаешь не сразу» — фото это комната со столом и
шкафом, история про скрытые удобства (подсветка зеркала на дверце,
выдвижной мини-стол, ящик для принтера).

### Позиционирование «мы объясняем решения» + блок выгод (17.07.2026)

Маркетинговая доработка под ключевую отличительную черту мастерской: они
не просто показывают результат, а **объясняют, почему предложено именно
такое решение** (видно и по странице проекта Сердце Столицы, и по стилю
общения). Эта идея сделана сквозной осью страницы, а не отдельной
вставкой:

* **Блок конструкции** (`.l-district`, секция «Из чего складывается
  хороший результат») несёт позиционирующее утверждение — класс
  `.l-manifesto` (serif italic, токены `--fs-md`/`--ink`): «Мы считаем,
  что хорошая мебель должна не только красиво выглядеть на фотографии, но
  и быть удобной каждый день. Поэтому мы всегда объясняем, почему
  предлагаем именно такое решение.» Плюс абзац «сначала конструкция,
  потом фасады». Манифест намеренно НЕ в самый верх страницы (абстракция
  до конкретики отпугивает) — он после hero, в блоке конструкции.
* **Подводка к ценам** (в том же блоке): «Мы работаем для тех, кому важна
  не самая низкая цена, а мебель, которая служит долго и удобна каждый
  день. Ориентир по цене такой: …» — мягкая квалификация сегмента без
  слов «премиум/дорого», отсеивает тех, кто ищет самое дешёвое.
* **Блок выгод** (`.l-utp`, тёмный) переписан с характеристик на выгоды
  клиента, заголовок «Почему с нами спокойно работать». 5 пунктов: цена
  не меняется после договора · 3D до производства · не нужно искать
  нескольких подрядчиков · мебель под ваше помещение · остаёмся на связи
  после установки. Икон не добавляли — переставили существующие 5 под
  новый смысл.
* **Интро портфолио** переписано под истории («мы показываем не только
  готовую мебель…»).
* **Блок «Не обещаем идеально. Объясняем, как будет на самом деле.»**
  (`.l-district`, светлый, после секции процесса) — честность ожиданий:
  если идея неудобна/дорога, предложим альтернативу и объясним почему.

Весь текст этих блоков — формулировки Александра (согласованы в сессии);
Claude только разложил их по местам и сверстал по дизайн-системе.

## SEO-доработки (11.07.2026)

Повод — отчёт стороннего SEO-чекера. Большинство его замечаний оказались
ложными (H1 присутствует на всех страницах, «встроенный CSS» — шум, короткий

**1. Канонизация URL — чистые адреса стали основными.**

* `.htaccess`: 301-редиректы `/страница.php → /страница`, `/index.php → /`,
`/abc/термин.html → /abc/термин` и срезание хвостового слеша у не-директорий.
Редиректы построены на `THE_REQUEST` (не `REQUEST_URI`), чтобы срабатывать
только на прямые запросы браузера и не зацикливаться с внутренними рерайтами.
* **`send.php` исключён из редиректов** — его вызывают формы POST'ом через
`fetch('send.php')`, 301 сломал бы отправку заявок. Не убирать исключение.
* **CORS (14.07.2026):** в `send.php` заголовок `Access-Control-Allow-Origin`
изменён со звёздочки `*` на `https://stivmebel.ru` — обработчик форм вызывают
только страницы своего сайта. Запросы с того же домена CORS не проверяют,
формы работают как раньше.
* **`send.php` возвращён в git (14.07.2026):** убран из `.gitignore` —
секретов в нём нет (всё в `config.php`), а раньше он выпадал из
GitHub-страховки, ради которой репозиторий и заводился. В `.gitignore`
остаются: `abc/*.html`, `abc/*.json`, `bbooks/`, `gallery/`, `video/*.mp4`,
`video/*.jpg`, `sitemap*.xml`, `*.log`, `.env`, `config.php`.
* Все внутренние ссылки переведены на чистые URL (header, footer, страницы,
лендинги, 404, генератор статей). Ссылка «Контакты» строится как
`($basePath ?: '/') . '#contact'`.
* `build.php`: sitemap генерируется с чистыми URL, `$canonical_path` статей
Азбуки — без `.html`, ссылки prev/next и «назад» внутри статей — чистые.

статьи и sitemap.
* Ссылка в Яндекс.Директе (`/land-ng`) уже была чистой — кампании не затронуты.
* **Конфликт имён страница/папка:** `/video` совпадает с папкой `video/`
(ролики), `/gallery` — с папкой `gallery/` (фото). Общий рерайт требует
«не директория», поэтому без явных правил Apache добавлял слеш и отдавал 403,
а первая попытка чинить это редиректом `/video/ → /video` дала петлю с

невозможна ни при каком поведении хостинга): в `.htaccess` — `DirectorySlash
Off` (если хостинг вернёт 500 на директиву, строку можно удалить, схема

`/video` и `/video/`.** `/gallery` и `/gallery/` — 301 на `/catalog` (цель не
директория, петли нет). Прямые ссылки на файлы (`/video/0001.mp4`,
`/gallery/0002-kitchen.jpg`) не затрагиваются. **При добавлении новой страницы
не называть её именем существующей папки** (images, assets, testimonials, abc)
