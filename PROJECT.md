# Стив Интерьеры — описание проекта

## Что это
Сайт-визитка мебельной мастерской (Станислав, Москва). Корпусная мебель на заказ.
Домен: **stivmebel.ru** (не stivinteriors.ru — это старый Tilda-сайт, не трогать).
AmoCRM: stivinteriors.amocrm.ru — базовый тариф, API недоступен, не использовать.

---

## Стек
- **PHP** — шаблонизация через `include 'header.php'` / `include 'footer.php'`
- **Vanilla JS** — без фреймворков
- **CSS custom properties** — единая дизайн-система
- Хостинг: **Reg.ru** shared. `fsockopen` к внешним SMTP заблокирован. `curl` к внешним хостам работает. `mail()` принимает письма но они не доходят (нет DKIM). Используем Telegram Bot API.

### config.php — единый источник важных данных

**Все чувствительные и часто меняемые настройки хранятся только в `config.php`.** Этот файл подключается через `require_once __DIR__ . '/config.php'` в начале каждой страницы и в `send.php`.

Что хранится в config.php:
- `SITE_PHONE`, `SITE_PHONE_HREF` — номер телефона (меняйте только здесь)
- `SITE_EMAIL`, `SITE_EMAIL_HREF` — email
- `SOCIAL_TG`, `SOCIAL_VK`, `SOCIAL_MAX` — ссылки на соцсети (иконки в футере и попап-форме)
- `SOCIAL_VK_PAGE`, `SOCIAL_TG_CHANNEL`, `YANDEX_MAPS_URL` — ссылки на страницы/каналы и карту (текстовые ссылки в футере и блоке «Контакты»)
- `SITE_NAME`, `SITE_TAGLINE` — название и слоган
- `YANDEX_METRIKA_ID` — ID счётчика Яндекс.Метрики
- `TG_TOKEN`, `TG_CHAT_ID` — токен Telegram-бота и ID чата получателя заявок

**Важно:** не хардкодить контакты, токены или ID напрямую в страницах — всё через константы из config.php.

Медиафайлы:
- `images/` — фото для index (k01–k08.jpg, m01–m06.jpg)
- `gallery/` — фото галереи (`0001-kitchen.jpg` … `0080-hallway.jpg`, формат `NNNN-category`)
- `video/` — видео: `NNNN.mp4` + `NNNN.jpg` (обложка) + опционально `NNNN.md` (подпись).
  Список читается автоматически через `glob()` в `video.php` — без `.md` видео
  показывается без подписи. См. раздел «Видео» ниже.
- `hello.mp4` — видео Станислава для about (в круге)

### Принцип разделения
| Файл | Что содержит |
|---|---|
| `style.css` | reset, переменные, анимации, nav, кнопки, секции, форма, footer, popup, адаптив |
| `aboutstyle.css` | hero с видео-кругом, why-grid, timeline, cta-block, results-grid |
| `gallerystyle.css` | сетка фото, фильтры, inquiry-bar, lightbox, modal + стили полей модала |
| `land-ng.css` | все стили лендинга: hero, district, utp, works, process, cta, quiz-модал (lq-) |
| `videostyle.css` | тёмная тема nav, video-feed, video-item, контролы плеера |

**Важно:** `gallerystyle.css` дублирует стили `.modal-field input/textarea` из `style.css` с `!important` — намеренно, иначе браузерные дефолты перебивают поля рамками.

Адаптив — основной брейкпоинт: `@media (max-width: 900px)`. Дополнительно в `gallerystyle.css` и `favoritesstyle.css` есть `@media (max-width: 500px)` — осознанное исключение для переключения сетки фото с 2 колонок на 1 на очень узких экранах, а не случайный третий брейкпоинт.

### Цепочки подключения и риск переопределений

Файл позже в цепочке побеждает при равной специфичности селектора — это
нормальный каскад CSS, но именно на стыке файлов рождаются «тихие» баги:
страничный файл переопределяет правило из общего `style.css`, и
переопределение либо теряет смысл исходного значения, либо расходится по
брейкпоинтам с третьим файлом в цепочке.

| Страница | Цепочка |
|---|---|
| index.php | style.css |
| about.php | style.css → aboutstyle.css |
| gallery.php | style.css → gallerystyle.css |
| favorites.php | style.css → gallerystyle.css → favoritesstyle.css |
| video.php | style.css → videostyle.css |
| price.php | style.css → pricestyle.css |
| abc.php / abc/*.html | style.css → abcstyle.css |

`favorites.php` — единственная страница с цепочкой длиной три файла, то есть
с наибольшим риском такого конфликта. Реальный случай: `favoritesstyle.css`
на мобильном переопределял `.fav-header { padding: 3rem ...}`, обнуляя
`padding-top: 7rem`, который `.gallery-header` в `gallerystyle.css` намеренно
ставил под высоту `position: fixed` навигации (`nav` в `style.css`) — в
результате шапка перекрывала заголовок страницы. Это и есть целевой класс
багов для процедуры проверки ниже (см. чеклист «CSS — консистентность»).

## Квиз (land-ng.php)

Встроенный квиз-модал, открывается кнопкой «Рассчитать стоимость» в герое.

Ответы собираются в `comment`, отправляются в `send.php` с `source='quiz'`.
В Telegram приходит с иконкой 🎯 и пометкой «Квиз».

## send.php — отправка заявок

**Telegram Bot API через curl.**

```
Bot token: см. BotFather
Chat ID:   см. BotFather (@mrdronru — Alexandr Dranitskiy, владелец)
```

Принимает JSON: `name`, `phone`, `type`, `comment`, `photos[]`, `source`.
Возвращает `{"ok":true}` или `{"ok":false,"error":"..."}`.

Поле `source`: `main` (форма index), `popup` (попап), `gallery` (модал галереи).

**`parse_mode` не используется — сообщения отправляются как обычный текст.** Раньше использовался `Markdown` с неполным экранированием (`tg_escape`), из-за чего спецсимволы в именах/телефонах/комментариях могли ломать запрос к Telegram API. Теперь `parse_mode` убран из вызова API целиком, разметка `*жирный*` не используется — пользовательский ввод безопасен в любом виде.

```
video/           папка
├── NNNN.mp4     видеофайл (обязательно)
├── NNNN.jpg     обложка/постер (обязательно)

```

Список видео в `video.php` читается автоматически через `glob('video/*.mp4')`
при каждой загрузке страницы — добавлять файлы вручную в код не нужно.
Сортировка — по имени файла (`0001`, `0002`...), как и раньше.

Если рядом с `NNNN.mp4` нет `NNNN.md` — видео всё равно показывается в ленте,
просто без заголовка и описания (блок `.video-meta` не выводится).

### Формат NNNN.md
```
---
title: Название видео
---

Текст описания в одну-несколько строк.
```

Описание выводится как обычный текст без markdown-разметки (жирный/курсив/
ссылки сознательно не поддерживаются здесь — `video.php` пропускает результат
`parseMarkdown()` через `strip_tags()`). Формат файла такой же простой, как у
заголовка — поддерживается только поле `title` в frontmatter.

**Статус:** подписи `0001.md`–`0020.md` подготовлены и должны лежать в `video/`
рядом с соответствующими `.mp4`/`.jpg`. При добавлении видео №21 и далее нужно
просто положить новый `NNNN.mp4` + `NNNN.jpg` (+ опционально `NNNN.md`) — список
на странице обновится сам.

### Генерация страниц
```bash

```
Запускать после каждого добавления или изменения MD-файлов.

при следующем запуске `build.php`, без необходимости синхронизировать вручную.
Используется `$basePath = '../'` (страницы лежат в подпапке `abc/`) и явный
`$canonical_path` для `og:url`/`canonical` — при генерации из CLI
`$_SERVER['REQUEST_URI']` не определён.

## Рефакторинг: переход на статический сайт

**Статус: только обсуждается, не начато.** Зафиксировано здесь, чтобы не потерять идею
и детали к моменту, когда решим приступить.

### Идея

PHP-страницы сейчас рендерятся заново при каждом обращении к сайту — это лишняя
нагрузка там, где контент не меняется от запроса к запросу. Предлагается по
аналогии с Азбукой (`build.php` уже генерирует `abc/*.html` из `.md` через
`header.php`/`footer.php`) расширить этот подход на весь сайт: превратить
контентные страницы в статический HTML, оставив PHP только там, где он
объективно нужен — серверная логика и секреты.

### Что остаётся на PHP всегда

- `send.php` — обработка формы, отправка в Telegram (токен бота, server-side по определению).
- `config.php`, `build.php`, `markdown-helpers.php` — служебные файлы, не отдаются напрямую.
- Любая логика, которая обязана выполняться на сервере при каждом запросе.

### Что переводится в статику

`index.php`, `about.php`, `price.php`, `gallery.php`, `video.php`, `favorites.php`,
`abc.php` (сам список терминов, не только сгенерированные статьи), `politika.php`,
`soglasie.php` — весь контентный слой сайта.

### Решённые вопросы (на этапе обсуждения)

- **Переход одним рефакторингом**, не постранично — весь сайт за один заход.
- **URL без `.html`** — красивые адреса (`/about`, `/gallery`) через `.htaccess`,
  не показывать расширение пользователю.
- **Пересборка после правок — нормальная практика**, уже привычна по работе
  с Азбукой: после любого изменения `header.php`/`footer.php`/другой общей части
  нужно заново запускать `build.php`, чтобы изменения попали на все страницы.
  Это осознанно принятый компромисс, не недостаток подхода.

### Технические задачи, которые предстоят

1. **Маршрутизация в `.htaccess`.** Нужно правило вида «если запрошен `/about`
   и существует `about.html` — отдать его», а не просто редирект. Более сложная
   логика, чем нынешний `.htaccess` (сейчас там только блокировка `config.php`).

2. **Переписать все внутренние ссылки сайта.** Сейчас везде `href="about.php"`,
   `href="gallery.php"` и т.д. — в `header.php`, `footer.php`, на каждой отдельной
   странице. При переходе на чистые URL все ссылки нужно переписать одним проходом.

3. **301-редирект со старых `.php`-адресов.** Текущие адреса (`stivmebel.ru/about.php`)
   уже какое-то время живут и вероятно частично проиндексированы — переезд URL без
   редиректа теряет SEO-вес (та же логика, что и в истории с `stivinteriors.ru`,
   см. историю переноса домена — нет отдельного раздела, но сама ситуация
   показательна: смена адреса без правильного 301 создаёт проблемы).

4. **`gallery.php` и `video.php` — главное изменение в рабочем процессе.**
   Сейчас обе страницы читают список файлов через `glob()` при каждом заходе —
   это и есть вся их «логика». В статике это означает: добавили фото/видео на
   сервер → нужно зайти на `build.php` → страница пересоберётся. Раньше было
   «просто залил файл — он сразу на сайте». Владелец предупреждён и согласен
   на этот компромисс.

5. **`favorites.php`.** Список избранного хранится в `localStorage` браузера —
   это уже клиентская логика, страница переносится в статику почти без проблем.

6. **`$canonical_path` для каждой страницы.** Та же история, что уже решена для
   Азбуки — при генерации из CLI `$_SERVER['REQUEST_URI']` не определён, нужно
   явно прописывать canonical/og:url для каждой генерируемой страницы.

7. **`build.php` должен научиться генерировать не только статьи Азбуки.**
   Сейчас генератор заточен под `abc/*.md → abc/*.html`. Нужно расширить его
   (или сделать параллельный генератор) для `index.html`, `about.html`,
   `gallery.html` и т.д. — каждая страница сохраняет свою текущую бизнес-логику
   (например, `glob()` для фото в галерее), просто результат уходит в файл,
   а не в ответ на HTTP-запрос.

### Перед началом работы — план первого шага

Когда будем готовы перейти от обсуждения к реализации: начать с инвентаризации —
собрать полный список всех файлов, всех внутренних ссылок и всех мест, которые
нужно изменить, прежде чем писать код. Масштаб рефакторинга подразумевает
системный проход по всем PHP-страницам, общим частям (`header.php`/`footer.php`),
`.htaccess` и всем внутренним ссылкам разом — стоит исключить риск что-то упустить.

## Чеклист аудита проекта

Список вопросов для проверки качества, консистентности и непротиворечивости кода.

### Структура и архитектура
- Все ли страницы используют единый `header.php` и `footer.php`?
- Нет ли страниц которые закрывают `</body></html>` вручную вместо `include footer.php`?
- Все ли CSS-файлы подключаются через `$extra_css` в header, а не вручную?
- Нет ли дублирования подключения `script.js` или других файлов?

### Конфигурация и хардкод
- Все ли контакты (телефон, email, соцсети) берутся из `config.php`, а не захардкожены в страницах?
- ID Яндекс.Метрики и токен Telegram — только в `config.php`?
- При изменении номера телефона — достаточно ли поменять только `config.php`?
- Список фотографий в `gallery.php` — читается из папки `gallery/` через `glob()`, а не хардкоженный массив?
- Список статей Азбуки — генерируется `build.php` из MD-файлов, а не вручную?

### CSS — консистентность
- Все ли цвета используют CSS-переменные (`var(--gold)`) а не хардкод (`#B8975A`)?
- Все ли отступы, шрифты, размеры берутся из дизайн-системы (`--fs-xs`, `--fs-hero` и т.д.)?
- Нет ли инлайн-стилей которые дублируют или переопределяют CSS-классы?
- Брейкпоинты соответствуют стандарту (`900px`, плюс осознанный `500px` — см. «Цепочки подключения» выше), нет ли случайных `768px` или `1024px`?
- Все ли страницы корректно отображаются на мобиле — нет ли горизонтального скролла?
- **Конфликты в цепочках подключения** (актуально прежде всего для `favorites.php`,
  у неё цепочка из трёх файлов — см. таблицу в «Архитектура CSS»): для каждого
  селектора, переопределяемого больше одного раза в цепочке одной страницы
  (включая правила внутри `@media`, не только верхнего уровня) — не теряет ли
  более позднее правило смысл значения из более раннего (особенно
  `padding-top`/`margin-top`, завязанные на высоту `position: fixed` навигации),
  и совпадают ли брейкпоинты между общим и страничным файлом?

### JavaScript — консистентность
- Нет ли дублирования одной и той же логики в разных файлах (маска телефона, валидация)?
- Все ли формы используют единую утилиту `window.formatPhone`?
- Все ли обработчики форм проверяют наличие элемента перед навешиванием (`if (el)...`)?
- Нет ли `querySelector` для элементов форм там где должен быть `getElementById`?
- Корректно ли работает `Escape` — закрывает только тот оверлей который открыт?

### Формы и отправка
- Все ли формы валидируют поля на клиенте перед отправкой?
- Все ли формы валидируются на сервере независимо от клиента?
- Показывается ли состояние загрузки на кнопке при отправке?
- Показывается ли success-состояние после успешной отправки?
- Обрабатывается ли ошибка сети — восстанавливается ли кнопка?
- Корректно ли `source` определяется в `send.php` для каждой формы?

### Мобильная версия
- Все ли секции имеют `padding: 4rem 1.5rem` на мобиле (единый ритм)?
- Нет ли элементов с фиксированной шириной в пикселях которые вылезают за экран?
- Нет ли `height: 100vh` без fallback — на iOS Safari это вызывает прыжок layout?
- Все ли кнопки в ряд переходят в колонку на мобиле?
- Корректно ли работает мобильное меню (drawer) на всех страницах?

### SEO и мета
- Есть ли уникальный `<title>` на каждой странице?
- Есть ли уникальный `<meta name="description">` на каждой странице?
- Есть ли Open Graph теги (`og:title`, `og:description`, `og:type`)?
- Есть ли `<meta name="robots" content="index, follow">`? (управляется через `$robots` в header.php; land-ng.php использует `noindex, follow`)
- Есть ли `lang="ru"` на `<html>`?

### Безопасность и надёжность
- `robots.txt` закрывает служебные страницы: favorites.php, 404.php, soglasie.php, politika.php, send.php, config.php, build.php
- Есть ли rate limiting на `send.php`?
- Валидируется ли телефон на сервере (минимум 10 цифр)?
- Нет ли чувствительных данных (токены, пароли) в публичном коде?
- Корректно ли стоит `session_start()` — до любых `header()` и вывода?

### Анимации и производительность
- Нет ли анимаций которые вызывают reflow (двигают элементы горизонтально)?
- Все ли вращающиеся/непрерывные анимации имеют `will-change: transform`?
- Все ли изображения имеют `loading="lazy"` или нативный lazy load?
- Нет ли `transform: translateY` на элементах без явной ширины — может вызвать горизонтальный сдвиг соседей?

### Доступность
- Все ли интерактивные элементы имеют `aria-label` если нет текста (иконки, соцсети)?
- Все ли изображения имеют осмысленный `alt`?
- Все ли модальные окна закрываются по `Escape`?
- Есть ли `<noscript>` на страницах где весь контент строится через JS?
