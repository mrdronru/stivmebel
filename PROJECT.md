# Стив Интерьеры — описание проекта

Главный файл проекта. Здесь общий слой, нужный в любой сессии, и карта, какой
файл читать под конкретную задачу. Документ разбит на части 21.07.2026, чтобы не
читать всё целиком ради одной темы.

## Карта документа

- **Правишь сайт (код, вёрстка, CSS, лендинги, SEO, формы, квиз, Азбука)** → `PROJECT-Сайт.md`.
- **Готовишь материалы и рекламу (каталог, презентация, КП, соцсети, Яндекс.Директ)** → `PROJECT-Продвижение.md`.
- **Ведёшь рассылки, переписку и продажи (дизайнеры, реактивация клиентов, схема сделки, скрипт общения)** → `PROJECT-Продажи.md`.

Новую информацию по теме записывай в соответствующий файл (по правилу «сначала
предложить, потом внести»), а сюда — только общепроектное: правила, контакты,
репозиторий, уроки работы с инструментами.

**Сквозные правила (действуют во всех частях):**
- Нигде не использовать длинное тире «—», только короткое «–» (глобальное правило проекта).
- Тон везде: конкретика вместо образов, честно про цену, без давления и искусственных дедлайнов.
- Размеры/цвета/значения — только из токенов `:root` (подробности в `PROJECT-Сайт.md`).
- Перекрёстные ссылки вида «см. раздел … ниже / в конце файла» после разбиения могут указывать на соседний файл части.

## Как ведётся этот документ

Как только в проекте происходит значимое изменение (новая архитектура, новый
файл/скрипт, изменение конвенции или принципа, важное решение с обоснованием),
которое может пригодиться позже для понимания проекта, — **это нужно сначала
предложить записать сюда**, и только после согласия действительно вносить
правку. Не молча и не «на автомате» — документ должен отражать то, что
реально согласовано, а не всё подряд, что происходило в сессии.

## Рабочие анти-паттерны Claude (уроки сессий)

**Правило: любой новый сбой или плохой паттерн в работе Claude фиксируется
здесь.** Если по ходу сессии обнаруживается глюк инструментов, повреждение
файлов, неудачный приём работы или ловушка окружения — это нужно записать в
этот раздел (кратко: симптом → причина → как избегать), чтобы следующие сессии
не наступали на те же грабли. Находка не должна теряться вместе с контекстом
сессии — это так же важно, как фиксировать содержательные решения.

### Повреждение файлов при смешивании путей записи (FUSE, найдено 17.07.2026)

Рабочая папка подключена как FUSE-мост между десктоп-приложением (хост) и
Linux-песочницей Claude. Записи идут двумя разными путями: инструменты
Read/Write/Edit — на стороне **хоста**, а `bash`/`python` — на стороне
**песочницы**.

**Симптом:** файл, который правился то Edit-инструментом, то bash/python-проходом
с коротким интервалом, повреждался — в хвост дописывались NUL-байты, либо файл
обрезался посреди строки. В сессии 17.07.2026 так обрезало `style.css` (потеряно
~46 строк хвоста) и дважды забивало NUL-ами конец `landing.css`. Системная
пометка «modified by a linter» — это просто харнесс замечал запись на стороне
хоста (собственный Edit Claude), стороннего линтера/вотчера нет.

**Причина:** рассинхрон FUSE при записи одного файла с двух сторон, особенно на
«сжимающей» записи (файл становится короче). Проверено контролируемым тестом:
чистые записи только через bash NUL/обрыва не дают ни разу; все инциденты шли
строго после Edit/Write-инструмента, переплетённого с bash по тому же файлу.

**Как избегать:**

* Для каждого файла держать **один путь записи за подход** — не перемежать
  Edit/Write-инструмент и bash/python по одному и тому же файлу.
* Массовые правки (регэксп-замены по многим файлам) делать **только через
  bash/python**; точечные правки — через инструменты, но без bash по этому же
  файлу рядом.
* После каждой записи проверять, что файл кончается ожидаемым символом (для CSS
  `}`) и не содержит NUL; при мусоре — вычистить (`rstrip(b'\x00')` + перевод
  строки).
* Перед массовыми правками держать бэкап затрагиваемых файлов.
* Удаление файлов в примонтированной папке из песочницы запрещено (`rm` →
  «Operation not permitted») — нужен инструмент удаления Cowork
  (`allow_cowork_file_delete`).

### Правка текста под видом фикса вёрстки (найдено 16.07.2026, land-mos)

**Симптом:** попросили исправить перенос строк в заголовке/абзацах land-mos
(текст «разъезжался» на 3 строки вместо 2). Вместо того чтобы решать это
размером шрифта/контейнера, Claude несколько раз подряд переписал сами
формулировки (сократил заголовок, переписал абзацы про материалы и цену) —
без запроса на изменение текста, только чтобы строки визуально влезли.

**Почему это плохо:** теряется утверждённый, согласованный смысл текста
(в данном случае — конкретная связка «точно подходит вашей квартире»,
важная для позиционирования лендинга). Правки текста вперемешку с правками
вёрстки трудно отследить и откатить: непонятно, где чинили строки, а где
незаметно сместился смысл.

**Правило на будущее:**

* Перенос строк, «крупно/мелко», «много воздуха» и подобные вёрсточные
  жалобы чинятся **только через CSS** (размер шрифта, line-height, ширина
  контейнера, паддинги) — никогда через сокращение или переписывание слов.
* При этом CSS-фикс делается **по правилам дизайн-системы** (см. «Правила
  дизайна — размеры и значения» выше): использовать ближайший подходящий
  токен из `:root`, не плодить новые произвольные значения (`clamp(...)`,
  случайные `rem`/`vw`) под одну страницу.
* Если после CSS-фикса строки всё равно не помещаются красиво — это повод
  спросить, можно ли сократить текст, а не переписывать его самостоятельно.

## GitHub

Код сайта (`/var/www/u2240000/data/www/stivmebel.ru`) — git-репозиторий,
подключённый к GitHub:

* **Репозиторий:** `git@github.com:mrdronru/stivmebel.git`
* **Ветка:** `main`

**Сохранение изменений — один скрипт.** На сервере, на уровень выше `www`
(там же, где сам `stivmebel.ru`), лежит исполняемый файл `stivm` — быстрый
коммит + пуш одной командой:

```bash
./stivm
```

(если `Permission denied` — один раз `chmod +x stivm`, дальше работает.)

Скрипт сам переходит в `SITE_DIR`, делает `git add .`, коммитит с
автосообщением вида `Изменение от ДД.ММ.ГГГГ ЧЧ:ММ` и пушит в `origin/main`.
Если изменений нет — просто сообщает об этом и ничего не коммитит.

**Важно: `./stivm` НЕ заливает файлы на сервер — это не деплой, а бэкап.**
Схема работы: локально лежит копия сайта (её правит Claude в подключённой
папке) → после порции правок Александр заходит на сервер, делает `./stivm`
(коммит+пуш текущего состояния сервера в git — точка отката/страховка) →
**затем вручную заливает изменённые файлы на сервер**. То есть деплой всегда
ручной; `./stivm` лишь фиксирует состояние перед ним. Claude не должен
называть `./stivm` «заливкой» и не должен предлагать его как способ
опубликовать правки.

**Именные версии (git-теги, добавлено 17.07.2026).** У `stivm` есть необязательный параметр — имя версии:

* `./stivm` — как выше: коммит с автосообщением + push.
* `./stivm <имя>` — коммит + **именная версия** (аннотированный git-тег `<имя>`), тег тоже пушится в origin. Если изменений нет, но имя задано — новый коммит не создаётся, тегом помечается текущее состояние (закладка).
* Если тег с таким именем уже существует — он НЕ перезатирается (это точка отката), скрипт выводит подсказку по удалению.
* Имя версии — латиницей/цифрами (пробелы и кириллица вычищаются в дефисы).
* Откат к именной версии: `git checkout <имя> -- .`; список всех версий: `git tag`.

Текущая проверенная база (после полного аудита сайта 17.07.2026) помечена тегом `check001`.

Посмотреть историю изменений:
```bash
cd /var/www/u2240000/data/www/stivmebel.ru
git log --oneline -10
```

**Почему это вообще завели:** однажды рекомендация Claude в одной из сессий
привела к стиранию актуальных файлов сайта, а восстановить было неоткуда —
в памяти/контексте той сессии их тоже уже не было. GitHub здесь не «для
галочки», а реальная страховка на именно такой случай. Из этого — практическое
правило на будущее: **перед любыми потенциально разрушительными действиями
(перезапись, массовое удаление, рефакторинг структуры файлов) сначала
`./stivm`**, чтобы был откат, и лишь затем вносить изменения.
