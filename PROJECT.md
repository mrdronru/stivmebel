### Повреждение файлов при смешивании путей записи (FUSE, найдено 17.07.2026)

Рабочая папка подключена как FUSE-мост между десктоп-приложением (хост) и
Linux-песочницей Claude. Записи идут двумя разными путями: инструменты
Read/Write/Edit — на стороне **хоста**, а `bash`/`python` — на стороне
**песочницы**.

**Симптом:** файл, который правился то Edit-инструментом, то bash/python-проходом
с коротким интервалом, повреждался — в хвост дописывались NUL-байты, либо файл
обрезался посреди строки. В сессии 17.07.2026 так обрезало `style.css` (потеряно
~46 строк хвоста) и дважды забивало NUL-ами конец `landing.css`. Системная
пометка «modified by a linter» — это просто харнесс замечал запись на стороне
хоста (собственный Edit Claude), стороннего линтера/вотчера нет.

**Причина:** рассинхрон FUSE при записи одного файла с двух сторон, особенно на
«сжимающей» записи (файл становится короче). Проверено контролируемым тестом:
чистые записи только через bash NUL/обрыва не дают ни разу; все инциденты шли
строго после Edit/Write-инструмента, переплетённого с bash по тому же файлу.

**Как избегать:**

* Для каждого файла держать **один путь записи за подход** — не перемежать
  Edit/Write-инструмент и bash/python по одному и тому же файлу.
* Массовые правки (регэксп-замены по многим файлам) делать **только через
  bash/python**; точечные правки — через инструменты, но без bash по этому же
  файлу рядом.
* После каждой записи проверять, что файл кончается ожидаемым символом (для CSS
  `}`) и не содержит NUL; при мусоре — вычистить (`rstrip(b'\x00')` + перевод
  строки).
* Перед массовыми правками держать бэкап затрагиваемых файлов.
* Удаление файлов в примонтированной папке из песочницы запрещено (`rm` →
  «Operation not permitted») — нужен инструмент удаления Cowork
  (`allow_cowork_file_delete`).

### Правка текста под видом фикса вёрстки (найдено 16.07.2026, land-mos)

**Симптом:** попросили исправить перенос строк в заголовке/абзацах land-mos
(текст «разъезжался» на 3 строки вместо 2). Вместо того чтобы решать это
размером шрифта/контейнера, Claude несколько раз подряд переписал сами
формулировки (сократил заголовок, переписал абзацы про материалы и цену) —
без запроса на изменение текста, только чтобы строки визуально влезли.

**Почему это плохо:** теряется утверждённый, согласованный смысл текста
(в данном случае — конкретная связка «точно подходит вашей квартире»,
важная для позиционирования лендинга). Правки текста вперемешку с правками
вёрстки трудно отследить и откатить: непонятно, где чинили строки, а где
незаметно сместился смысл.

**Правило на будущее:**

* Перенос строк, «крупно/мелко», «много воздуха» и подобные вёрсточные
  жалобы чинятся **только через CSS** (размер шрифта, line-height, ширина
  контейнера, паддинги) — никогда через сокращение или переписывание слов.
* При этом CSS-фикс делается **по правилам дизайн-системы** (см. «Правила
  дизайна — размеры и значения» выше): использовать ближайший подходящий
  токен из `:root`, не плодить новые произвольные значения (`clamp(...)`,
  случайные `rem`/`vw`) под одну страницу.
* Если после CSS-фикса строки всё равно не помещаются красиво — это повод
  спросить, можно ли сократить текст, а не переписывать его самостоятельно.

## GitHub

Код сайта (`/var/www/u2240000/data/www/stivmebel.ru`) — git-репозиторий,
подключённый к GitHub:

* **Репозиторий:** `git@github.com:mrdronru/stivmebel.git`
* **Ветка:** `main`

**Сохранение изменений — один скрипт.** На сервере, на уровень выше `www`
(там же, где сам `stivmebel.ru`), лежит исполняемый файл `stivm` — быстрый
коммит + пуш одной командой:

```bash
./stivm
```

(если `Permission denied` — один раз `chmod +x stivm`, дальше работает.)

Скрипт сам переходит в `SITE_DIR`, делает `git add .`, коммитит с
автосообщением вида `Изменение от ДД.ММ.ГГГГ ЧЧ:ММ` и пушит в `origin/main`.
Если изменений нет — просто сообщает об этом и ничего не коммитит.

**Важно: `./stivm` НЕ заливает файлы на сервер — это не деплой, а бэкап.**
Схема работы: локально лежит копия сайта (её правит Claude в подключённой
папке) → после порции правок Александр заходит на сервер, делает `./stivm`
(коммит+пуш текущего состояния сервера в git — точка отката/страховка) →
**затем вручную заливает изменённые файлы на сервер**. То есть деплой всегда
ручной; `./stivm` лишь фиксирует состояние перед ним. Claude не должен
называть `./stivm` «заливкой» и не должен предлагать его как способ
опубликовать правки.

**Именные версии (git-теги, добавлено 17.07.2026).** У `stivm` есть необязательный параметр — имя версии:

* `./stivm` — как выше: коммит с автосообщением + push.
* `./stivm <имя>` — коммит + **именная версия** (аннотированный git-тег `<имя>`), тег тоже пушится в origin. Если изменений нет, но имя задано — новый коммит не создаётся, тегом помечается текущее состояние (закладка).
* Если тег с таким именем уже существует — он НЕ перезатирается (это точка отката), скрипт выводит подсказку по удалению.
* Имя версии — латиницей/цифрами (пробелы и кириллица вычищаются в дефисы).
* Откат к именной версии: `git checkout <имя> -- .`; список всех версий: `git tag`.

Текущая проверенная база (после полного аудита сайта 17.07.2026) помечена тегом `check001`.

Посмотреть историю изменений:
```bash
cd /var/www/u2240000/data/www/stivmebel.ru
git log --oneline -10
```

**Почему это вообще завели:** однажды рекомендация Claude в одной из сессий
привела к стиранию актуальных файлов сайта, а восстановить было неоткуда —
в памяти/контексте той сессии их тоже уже не было. GitHub здесь не «для
галочки», а реальная страховка на именно такой случай. Из этого — практическое
правило на будущее: **перед любыми потенциально разрушительными действиями
(перезапись, массовое удаление, рефакторинг структуры файлов) сначала
`./stivm`**, чтобы был откат, и лишь затем вносить изменения.

## Стек

* **PHP** — шаблонизация через `include 'header.php'` / `include 'footer.php'`
* **Vanilla JS** — без фреймворков
* **CSS custom properties** — единая дизайн-система

**Правила дизайна — кнопки (зафиксировано 14.07.2026 после двух повторов
одной ошибки):** любая кнопка и CTA оформляются ТОЛЬКО сплошной заливкой
с контрастным текстом — никаких контурных/прозрачных кнопок и никогда цвет
кнопки не должен совпадать или сближаться с фоном блока. Канон: на светлом
фоне (cream/warm-white) — `.btn-primary` как есть: заливка `--ink`, текст
`--warm-white`; на тёмном фоне (`--ink`) — заливка `--gold`, текст `--ink`,
ховер `--gold-light` (пример: `.project-cta-btn` в projectstyle.css).
Стандартный `.btn-primary` на тёмном блоке ставить нельзя — чернила на
чернилах не читаются как кнопка. Перед сдачей любой страницы проверять
кнопку на каждом фоне, где она встречается.

**Правила дизайна — размеры и значения (дизайн-система, зафиксировано
17.07.2026).** Главная — эталон дизайн-системы. Все значения (размеры
шрифтов, прозрачности, трекинг, длительности переходов, цвета) берутся
только из токенов в `:root` `style.css` (полный список — в разделе
«CSS-переменные» ниже). Никакая страница (внутренние, лендинги, Азбука,
проекты) не должна вводить размер/цвет/значение, которого нет в системе.

*Как понимать относительные просьбы.* «Сделай крупнее / мельче / плотнее /
светлее / быстрее» и т.п. — это НЕ повод писать произвольное число. Это
значит взять **ближайшую подходящую ступень из уже существующей шкалы**:
соседний `--fs-*`, соседнюю альфу `--ink/ww/gold-XX`, соседний `--ls-*` или
`--dur-*`.

*Если подходящего токена в системе нет* — не заводить значение молча.
Сообщить об этом и вместе с Александром выбрать один из двух путей:
(1) добавить новую ступень в `:root` — если это осмысленное общее значение,
которое пригодится и дальше; либо (2) сделать **разовый литерал прямо в
страничном CSS второго уровня** (`aboutstyle`/`landing`/…) — если значение
нужно ровно один раз и в шкалу не просится (такие one-off в `:root` не
выносим). Новый токен в `:root` заводится только под реальный кластер
употреблений и только с одобрения Александра.
* Хостинг: **Reg.ru** shared. Для форм сайта (`send.php`): `fsockopen` к внешним SMTP заблокирован, `curl` к внешним хостам работает, `mail()` принимает письма, но они не доходят (нет DKIM на уровне PHP-отправки) — поэтому заявки с форм идут в Telegram Bot API, а не на почту. Это ограничение не относится к отдельному процессу массовой рассылки дизайнерам (Python `smtplib` через SSH — см. раздел «B2B-рассылка дизайнерам» в конце файла), там SMTP настроен и работает.

### config.php — единый источник важных данных

## Яндекс.Метрика — цели

Хелпер `ymGoal(name)` определён в `footer.php` и доступен глобально.
Все вызовы через `ymGoal(...)`, не напрямую через `ym(...)`.

|Идентификатор|Событие|Где срабатывает|
|-|-|-|
|`click\_phone`|Клик по телефону|header, footer, index, about, price, favorites, land-ng, land-cc, land-mos|
|`click\_telegram`|Клик Telegram|header, footer, index|
|`click\_vk`|Клик ВКонтакте|header, footer, index|
|`form\_submit`|Отправка формы|popup (footer.php), главная форма (script.js)|
|`quiz\_open`|Квиз открыт|любой лендинг с квизом через `quiz.php` (land-ng.php, land-mos.php; на land-cc квиза нет)|
|`quiz\_step\_4`|Квиз: шаг 4|—//— (условие в JS `current >= 4`, шаги 1–3 цель не шлют)|
|`quiz\_step\_5`|Квиз: шаг 5|—//—|
|`quiz\_step\_6`|Квиз: шаг 6|—//—|
|`quiz\_complete`|Квиз завершён|—//—|
|`hero\_video\_play`|Открытие видео со звуком в попапе|land-cc.php (кнопка «▶ Видео со звуком» в hero)|

**Цели в коде vs. настройки счётчика (найдено 18.07.2026).** `ymGoal(...)`
(→ `reachGoal`) отправляет событие всегда, но в отчётах Метрики оно считается
только если в счётчике заведена цель типа «JavaScript-событие» с
идентификатором, **точно совпадающим** с тем, что шлёт код. При проверке квиза
на land-mos нашли рассинхрон: в счётчике была лишняя цель `quiz_step_3`
(никогда не срабатывала — код шлёт шаги только при `current >= 4`, см. ниже) и
не было `quiz_step_6` (код её шлёт, но событие уходило в никуда). Исправлено на
стороне Метрики: у существующей цели `quiz_step_3` поменяли **идентификатор**
(не только подпись — считается по идентификатору) на `quiz_step_6`.

Актуальный набор целей в счётчике 108718941 после правки: `quiz_open`,
`quiz_step_4`, `quiz_step_5`, `quiz_step_6`, `quiz_complete` (+ `click_phone`,
`form_submit` и т.д.). Воронка квиза: открыт → шаг 4 → шаг 5 → шаг 6 →
завершён. Шаги 1–3 и экран контактов (шаг 7) своих целей не имеют — если
понадобится мерить отвал на форме, добавить цель на показ шага 7 в `quiz.php`.

Как проверить, что цель уходит: DevTools → Network → запрос на
`mc.yandex.ru/watch/<id>` с `page-url=goal://…/<идентификатор>` и
`page-ref=` страницы; ответ 302→200 — это штатный редирект Метрики, а не ошибка.
Запрос на `mc.yandex.ru/clmap/…` — это карта кликов, не цель.

## Квиз — общий файл quiz.php (рефакторинг 16.07.2026)

Раньше у каждого лендинга была своя копия квиза прямо в разметке страницы —
land-ng и land-mos успели разойтись (разное число шагов, разные вопросы,
разные способы связи), и точно так же разошлись бы любые следующие
лендинги. Теперь квиз — один файл `quiz.php`, подключается через `include`
там, где нужен:

```php
<?php $quiz_source = 'quiz-ng'; include 'quiz.php'; ?>
```

`$quiz_source` — обязательная переменная, задаётся ПЕРЕД `include`, уходит в
`send.php` как `source` (например `quiz-ng`, `quiz-mos`) — по ней видно, с
какого именно лендинга пришла заявка, и в Telegram, и в Метрике. Если не
задать — используется `'quiz'` (не ломается, но обезличено).

Сейчас подключён на `land-ng.php` (`source: 'quiz-ng'`) и `land-mos.php`
(`source: 'quiz-mos'`). На `land-cc.php` квиза по-прежнему нет.

**Важно:** `quiz.php` — единственный источник правды для вопросов и логики
квиза. Менять шаги/вопросы/способы связи нужно только там — правка сразу
применяется на всех лендингах, которые его подключают. Не копировать
разметку квиза обратно в конкретный `land-*.php`.

### Шаги (7 содержательных + экран входа + экран «спасибо»)

0. Экран входа — кнопка «Начать» (`startQuiz()`)
1. Что хотите сделать? — Кухню / Шкаф / Гардеробную / Несколько предметов мебели
2. Для какого помещения нужна мебель? — **пропускается автоматически**, если
   на шаге 1 выбрали «Кухню» (ответ подставляется как «Кухня», переход сразу
   на шаг 3; кнопка «Назад» с шага 3 в этом случае ведёт на шаг 1, а не на
   шаг 2 — флаг `roomSkipped` в JS)
3. Какой примерно размер помещения?
4. Какая задача сейчас самая важная?
5. На какой ориентир по стоимости рассчитываете? (5-й вариант — «Пока
   сложно оценить»)
6. Когда планируете заказать мебель?
7. Контакт — имя + способ связи: Телефон, WhatsApp, Telegram, Макс, Email

Экран «спасибо» — `data-step="8"`.

### send.php и source

`send.php` определяет заявку с квиза по префиксу: `strpos($source, 'quiz')
=== 0` — значит `quiz`, `quiz-ng`, `quiz-mos` и любой будущий
`quiz-<новый-лендинг>` обрабатываются одинаково (пропуск проверки
«похоже на телефон» для нетелефонных способов связи вроде Telegram/Email).
Подписи в Telegram берутся из массива `$quizSourceLabels` в `send.php` — при
подключении квиза на новый лендинг туда стоит добавить строку
(`'quiz-<название>' => 'Квиз (...)'`), иначе в Telegram будет просто «Квиз»
без уточнения, с какой страницы.

**Найденный и исправленный баг:** до этого рефакторинга сравнение было
`$source === 'quiz'` дословно — `land-mos.php` уже отправлял `'quiz-mos'`,
из-за чего для Telegram/Email в квизе на land-mos срабатывала проверка
«номер телефона обязателен», и заявка тихо отклонялась (без видимой ошибки
у пользователя, просто кнопка «Попробовать ещё раз»). Исправлено вместе с
переносом квиза в общий файл.

## send.php — отправка заявок

**Telegram Bot API через curl.**

```
Bot token: см. BotFather
Chat ID:   см. BotFather (@mrdronru — Alexandr Dranitskiy, владелец)
```

Принимает JSON: `name`, `phone`, `type`, `comment`, `photos\[]`, `source`.
Возвращает `{"ok":true}` или `{"ok":false,"error":"..."}`.

Поле `source`: `main` (форма index), `popup` (попап), `gallery` (модал галереи), `project-cc` (попап со страницы проекта; в Telegram — «Страница проекта (Сердце Столицы)»). Механика: попап в footer.php берёт источник из `window.popupSource || 'popup'` — страница проекта задаёт `window.popupSource='project-cc'` перед подключением футера; новые страницы раздела делают так же со своим кодом.

**Квиз — источники вида `quiz-<лендинг>`.** С 16.07.2026 квиз общий (`quiz.php`, см. одноимённый раздел), и `source` приходит не просто `quiz`, а с суффиксом лендинга — `quiz-ng`, `quiz-mos` и т.д. `send.php` матчит такие значения по префиксу (`strpos($source, 'quiz') === 0`), а не точным сравнением — это и определяет, пропускать ли проверку «похоже на телефон» для нетелефонных способов связи (Telegram/Email/и т.п.). Подписи для Telegram-сообщения берутся из массива `$quizSourceLabels`; при подключении квиза на новый лендинг нужно добавить туда строку `'quiz-<название>' => 'Квиз (...)'`, иначе в сообщении будет просто «Квиз» без уточнения страницы.

**`parse\_mode` не используется — сообщения отправляются как обычный текст.** Раньше использовался `Markdown` с неполным экранированием (`tg\_escape`), из-за чего спецсимволы в именах/телефонах/комментариях могли ломать запрос к Telegram API. Теперь `parse\_mode` убран из вызова API целиком, разметка `\*жирный\*` не используется — пользовательский ввод безопасен в любом виде.

### Картинки в статьях (добавлено 15.07.2026)

До статьи `ergonomika-kuhni` (первая статья с картинками, letter «Э») в
Азбуке не было ни одной иллюстрированной статьи — парсер и структура папок
были рассчитаны только на текст. Конвенция:

* **Хранение:** `abc/images/<slug>/` — своя подпапка на каждую статью с
  картинками, по её уникальному slug. Не пересекается с `gallery/`,
  `images/`, `projects/`, `testimonials/`, `video/`. В `.gitignore` строка
  `abc/images/` — картинки статей, как и остальные медиапапки, не хранятся
  в git, заливаются на сервер напрямую.
* **Markdown:** `markdown-helpers.php` → `inlineMarkdown()` поддерживает
  `![alt](путь)`. Regex картинки стоит ОБЯЗАТЕЛЬНО до regex ссылки
  `[текст](ссылка)` — у них общий "хвост" `(...)`, при обратном порядке
  ссылка перехватывает картинку раньше и превращает её в битую ссылку.
  Путь в `![]()` указывается относительно `abc/`, например
  `images/ergonomika-kuhni/01-vytyazhka.jpg` — резолвится корректно от
  чистого URL `/abc/<slug>` (та же логика, что у ссылок «пред/след»).
* **Стили:** `abcstyle.css` → `.abc-article-body img` (картинка рендерится
  парсером как `<p><img></p>`, стиль без обёртки).

**Паттерн «иллюстрация + текст-панель»** — если источник картинки сырая
инфографика с текстом внутри самой картинки: НЕ кропить текстовый блок как
есть — шрифт и отступы исходника почти никогда не совпадают с сеткой кропа
(текст обрезается с одного края или остаётся лишний воздух с другого).
Вместо этого кропится только иллюстрация, а короткий текст рядом рендерится
заново (Pillow, `ImageDraw` + `ImageFont`, шрифт `LiberationSans-Bold.ttf`,
фон `--cream` `#F5F2EC`, золотая черта-акцент `--gold` `#B8975A` сверху,
авто-подбор размера шрифта под длину текста). Даёт предсказуемый результат
без ручной подгонки кропа под каждый текст.

### Апскейл иллюстраций — рецепт для песочницы (добавлено 15.07.2026)

Песочница Claude стартует с нуля в каждой сессии — если апскейл понадобится
снова, вот рабочий рецепт (проверено на статье `ergonomika-kuhni`):

* Пакет `realesrgan-ncnn-py` с PyPI — модели уже внутри wheel (~45 МБ), не
  нужно тянуть веса отдельно (GitHub releases/raw.githubusercontent.com/
  huggingface.co в песочнице заблокированы allowlist'ом сети).
* Не хватает системных библиотек `libvulkan.so.1` и `libomp.so.5` — root
  недоступен, `apt-get install` не работает. Решение: скачать `.deb` пакеты
  напрямую с `archive.ubuntu.com` (доступен без ограничений) —
  `vulkan-loader` (`libvulkan1_*.deb`) и `llvm-toolchain-14`
  (`libomp5-14_*.deb`), распаковать через `dpkg-deb -x <файл> <папка>` (не
  требует root), путь до `.so`-файлов добавить в `LD_LIBRARY_PATH`.
* `Realesrgan(gpuid=-1, model=3)` — `gpuid=-1` включает CPU-режим (в
  песочнице нет GPU/Vulkan-устройства, но сам загрузчик `libvulkan.so.1`
  всё равно должен физически присутствовать, иначе `.so`-обёртка не
  импортируется). `model=3` — `realesrgan-x4plus-anime`, лучше подходит для
  плоской line-art графики (чистые контуры, текст на иллюстрации), чем
  фото-модель `model=4` (`realesrgan-x4plus`).
* ~7–15 сек на одну картинку на CPU.

## Лендинг land-mos.php — архитектура и решения

Третий лендинг, для широкой аудитории Москвы (группа 3 в кампании №4, копия
настроек группы 2, геотаргетинг — вся Москва). Делит общий `landing.css` и
шапку `.ln-nav` с land-ng/land-cc, но своя аудитория и тональность: люди,
которые понимают, что «хорошо и персонально» стоит дороже, чем «как
получится», но не ищут слово «премиум» — без «премиум/элит/роскошь» в
текстах, конкретика вместо образов (материалы, гарантии, сроки).

### Кнопка «Написать нам» вместо WhatsApp — общий дизайн-принцип

В hero и в финальном CTA кнопка ведёт не на `wa.me`-ссылку, а открывает
попап-форму (`openContactPopup()`). Причина — в России WhatsApp и Telegram
работают только через VPN, это неудобно и есть не у всех: прямая
CTA-кнопка на мессенджер отсекает часть аудитории уже на клике. Актуально не
только для этого лендинга — при создании новых страниц с CTA на мессенджер
по умолчанию стоит предлагать форму, а не прямую ссылку на чат.

### Цены — динамически из config.php

Текст про цены не хардкодит цифры, а вызывает `price_from('kitchens')` /
`price_from('wardrobes')` / `price_from('closets')` — все три цены (кухни,
шкафы, гардеробные) берутся из `PRICES` и показываются единообразно «от N ₽
за п.м.». При изменении цен в `PRICES` текст на странице обновится сам, без
правки land-mos.php. (Кухня раньше была захардкожена «от 300 000 ₽» — заменено
17.07.2026 на `price_from('kitchens')`, за п.м., как остальные.)

`price_from()` вставляет внутрь цены неразрывные пробелы U+00A0 (и в
разделитель тысяч, и перед ₽), чтобы «от 59 500 ₽» переносилось только
целиком — ни «59 500», ни знак ₽ не съезжают на другую строку. Используется
реальный символ `"\xC2\xA0"`, а НЕ entity `&nbsp;`: строка идёт и в
meta-description (`price.php`), где entity экранировалась бы и показалась
буквально. Правка глобальная — действует на всех страницах, где выводится
цена (land-mos, главная, страница цен). Добавлено 17.07.2026.

### Портфолио — истории проектов (переделано 17.07.2026)

Те же 5 фото: `gallery/0002-kitchen.jpg`, `0073-hallway.jpg`,
`0091-kitchen.jpg`, `0109-kitchen.jpg`, `0056-wardrobe.jpg`. Но вместо
плотной фото-мозаики с однострочными подписями (как на land-ng/land-cc) —
**просторные карточки**: фото + заголовок + короткая история проекта в
2–3 предложения по схеме «задача → решение → почему именно так». Это
операционализирует сквозную идею лендинга (см. ниже «мы объясняем
решения»): портфолио не показывает, а рассказывает.

Карточный вид сделан модификатором `.l-portfolio--stories` на секции
работ (НЕ на секции отзывов, у неё тот же класс `.l-portfolio`) —

`.l-pf-*`, которую делят land-ng и land-cc (проверено: у них 5 работ

### Позиционирование «мы объясняем решения» + блок выгод (17.07.2026)

Маркетинговая доработка под ключевую отличительную черту мастерской: они
не просто показывают результат, а **объясняют, почему предложено именно
такое решение** (видно и по странице проекта Сердце Столицы, и по стилю
общения). Эта идея сделана сквозной осью страницы, а не отдельной
вставкой:

* **Блок конструкции** (`.l-district`, секция «Из чего складывается
  хороший результат») несёт позиционирующее утверждение — класс
  `.l-manifesto` (serif italic, токены `--fs-md`/`--ink`): «Мы считаем,
  что хорошая мебель должна не только красиво выглядеть на фотографии, но
  и быть удобной каждый день. Поэтому мы всегда объясняем, почему
  предлагаем именно такое решение.» Плюс абзац «сначала конструкция,
  потом фасады». Манифест намеренно НЕ в самый верх страницы (абстракция
  до конкретики отпугивает) — он после hero, в блоке конструкции.
* **Подводка к ценам** (в том же блоке): «Мы работаем для тех, кому важна
  не самая низкая цена, а мебель, которая служит долго и удобна каждый
  день. Ориентир по цене такой: …» — мягкая квалификация сегмента без
  слов «премиум/дорого», отсеивает тех, кто ищет самое дешёвое.
* **Блок выгод** (`.l-utp`, тёмный) переписан с характеристик на выгоды
  клиента, заголовок «Почему с нами спокойно работать». 5 пунктов: цена
  не меняется после договора · 3D до производства · не нужно искать
  нескольких подрядчиков · мебель под ваше помещение · остаёмся на связи
  после установки. Икон не добавляли — переставили существующие 5 под
  новый смысл.
* **Интро портфолио** переписано под истории («мы показываем не только
  готовую мебель…»).
* **Блок «Не обещаем идеально. Объясняем, как будет на самом деле.»**
  (`.l-district`, светлый, после секции процесса) — честность ожиданий:
  если идея неудобна/дорога, предложим альтернативу и объясним почему.

### Идея

PHP-страницы сейчас рендерятся заново при каждом обращении к сайту — это лишняя
нагрузка там, где контент не меняется от запроса к запросу. Предлагается по
аналогии с Азбукой (`build.php` уже генерирует `abc/\*.html` из `.md` через
`header.php`/`footer.php`) расширить этот подход на весь сайт: превратить
контентные страницы в статический HTML, оставив PHP только там, где он
объективно нужен — серверная логика и секреты.

### Что остаётся на PHP всегда

* `send.php` — обработка формы, отправка в Telegram (токен бота, server-side по определению).
* `config.php`, `build.php`, `markdown-helpers.php` — служебные файлы, не отдаются напрямую.
* Любая логика, которая обязана выполняться на сервере при каждом запросе.

### Что переводится в статику

`index.php`, `about.php`, `price.php`, `catalog.php`, `video.php`, `favorites.php`,
`abcd.php` (сам список терминов, не только сгенерированные статьи), `politika.php`,
`soglasie.php` — весь контентный слой сайта.

### Решённые вопросы (на этапе обсуждения)

* **Переход одним рефакторингом**, не постранично — весь сайт за один заход.
* **URL без `.html`** — красивые адреса (`/about`, `/gallery`) через `.htaccess`,
не показывать расширение пользователю.
* **Пересборка после правок — нормальная практика**, уже привычна по работе
с Азбукой: после любого изменения `header.php`/`footer.php`/другой общей части
нужно заново запускать `build.php`, чтобы изменения попали на все страницы.
Это осознанно принятый компромисс, не недостаток подхода.

### Технические задачи, которые предстоят

1. **Маршрутизация в `.htaccess`.** ✅ Выполнено досрочно 11.07.2026 в рамках
SEO-доработок (см. раздел «SEO-доработки» ниже): чистые URL теперь канонические.
Для перехода на статику останется только поменять приоритет в правилах: если
существует `.html` — отдать его, иначе `.php` (на переходный период).
2. **Переписать все внутренние ссылки сайта.** ✅ Выполнено 11.07.2026: все
внутренние ссылки в `header.php`, `footer.php`, страницах, лендингах, `404.php`
и генераторе `build.php` переведены на чистые URL одним проходом.
3. **301-редирект со старых `.php`-адресов.** ✅ Выполнено 11.07.2026 —
см. раздел «SEO-доработки». Проиндексированные `.php`-адреса и `abc/*.html`
отдают 301 на чистые URL, SEO-вес переносится.
4. **`catalog.php` и `video.php` — главное изменение в рабочем процессе.**
Сейчас обе страницы читают список файлов через `glob()` при каждом заходе —
это и есть вся их «логика». В статике это означает: добавили фото/видео на
сервер → нужно зайти на `build.php` → страница пересоберётся. Раньше было
«просто залил файл — он сразу на сайте». Владелец предупреждён и согласен
на этот компромисс.
5. **`favorites.php`.** Список избранного хранится в `localStorage` браузера —
это уже клиентская логика, страница переносится в статику почти без проблем.
6. **`$canonical\_path` для каждой страницы.** Та же история, что уже решена для
Азбуки — при генерации из CLI `$\_SERVER\['REQUEST\_URI']` не определён, нужно
явно прописывать canonical/og:url для каждой генерируемой страницы.
7. **`build.php` должен научиться генерировать не только статьи Азбуки.**
Сейчас генератор заточен под `abc/\*.md → abc/\*.html`. Нужно расширить его
(или сделать параллельный генератор) для `index.html`, `about.html`,
`gallery.html` и т.д. — каждая страница сохраняет свою текущую бизнес-логику
(например, `glob()` для фото в галерее), просто результат уходит в файл,
а не в ответ на HTTP-запрос.

Когда будем готовы перейти от обсуждения к реализации: начать с инвентаризации —

# B2B-рассылка дизайнерам интерьера (Флатика)

## Цель

Найти на flatica.ru дизайнеров интерьера, которым может быть нужен подрядчик
по мебели на заказ, и написать им персонализированные холодные письма от
лица «Стив Интерьеры» (Александр Драницкий).

## Пайплайн (по шагам)

1. **Сбор данных** — `flatica\_designers.py`
2. **Очистка и разметка** — `designers\_analyzed.csv` → `designers\_filtered.csv` → `designers\_segmented.csv`
3. **Персонализация писем по сегменту** — `campaignN\_M.csv` (N — номер сегмента/пачки, M — число строк)
4. **Отправка** — `send\_campaign.py` + `env.sh` (на сервере хостинга)
5. **Учёт ответов** — вручную, в колонках `replied` / `reply\_notes` / `status` того же CSV

Эти файлы (скрипты, CSV) находятся отдельно от кода сайта — в рабочей папке
`/var/www/u2240000/data/flatica` на том же хостинге, но **вне** `data/www/`,
чтобы не раздаваться через веб-сервер (там реальные email-адреса дизайнеров).

## 1\. Скрапер — `flatica\_designers.py`

Собирает профили дизайнеров со страницы списка (например,
`https://flatica.ru/professionals/speciality.dizayn-interyera\~city.moskva`).

**Ключевая архитектура:**

* Список дизайнеров требует настоящего браузера (Playwright) — сайт грузит
карточки через JS с задержкой 5-8 сек после докрутки до конца страницы.
Скрипт докручивает до текущего конца → ждёт с опросом высоты страницы
(`LOAD\_WAIT\_MAX=10 сек`, `LOAD\_POLL\_INTERVAL=1 сек`) → повторяет, пока
`STALL\_LIMIT=3` докрутки подряд не дают роста.
* **Страницы профиля НЕ требуют кликов вообще** — flatica.ru рендерит всё на
сервере (SSR): биография, первая страница проектов и отзывов уже есть в
обычном HTML. Разбор идёт через `BeautifulSoup` по реальным id-якорям
(`id="projects"`, `id="reviews"`, `id="about"`), без единого клика по
вкладкам. Это было НЕ очевидно и стоило \~15 итераций отладки.
* Телефон достаётся из ссылки на WhatsApp (`wa.me/...`) в HTML — в контактах
он всегда замаскирован звёздочками и требует клика, но в WhatsApp-ссылке
уже в открытом виде.
* Имя берётся из `<h2>`, не `<h1>` — `<h1>` на странице flatica — задвоенная
карточка в подвале («Дмитрий КрассаДмитрий Красса»).
* Отфильтровываются собственные соцсети Флатики (footer), которые иначе
попадают в каждую строку — список в `FLATICA\_OWN\_LINK\_MARKERS`.
* Колонки: `name, url, city, phone, email, website, telegram, whatsapp, vk, dzen, youtube, instagram, other\_social, category, rating, review\_count, about\_text, projects, reviews\_excerpt`.
* Лог — одновременно в консоль и в `debug.log` (см. `\_Tee` в начале файла).
* Флаги: `--max-profiles N`, `--out FILE.csv`, `--headless=false`, `--debug`.

**Известное ограничение:** без клика на «Показать ещё» в CSV попадает только
первая страница портфолио/отзывов (обычно 4-6 штук) — осознанный компромисс
ради надёжности после долгой отладки кликов.

### Правила составления «лести» (зацепки) — накопленный опыт ручной вычитки

После первого автопрохода по всем 140 строкам выяснилось: у Claude нет
доступа к фотографиям на странице профиля Флатики, только к тексту. Из-за
этого большая часть автоматически сгенерированных зацепок оказалась не
деталью проекта, а фактом о компании (стаж, регалии, география, философия) —
и такие факты плохо стыкуются со стандартной концовкой «Видно, что решение
продиктовано не только эстетикой, но и тем, как этим будут пользоваться»,
которая по смыслу требует именно детали решения, а не общего факта.

**Рабочий процесс, который сложился:** Александр сам смотрит фото проекта на
Флатике и присылает Claude либо готовую деталь, либо описание/фото — Claude
формулирует по нему короткую зацепку. Это надёжнее, чем просить Claude
угадывать деталь по одному тексту.

**Правила формулировок, выведенные через правки:**
* Назвать одну конкретную деталь и **не разжёвывать**, почему она хороша —
короче почти всегда лучше.
* Различать деталь проекта (визуальная, конкретная — подходит под стандартную
концовку) и факт о компании (стаж, награды, позиционирование — под неё не
подходит, нужна другая связка или вообще без неё).
* Вопросы — **без отрицания** («Не нужен ли...» провоцирует ответ «не нужен»;
переформулировать в утвердительную форму: «Бывает нужен...»).
* Без расплывчатых обещаний («если когда-нибудь будет нужен» → конкретнее:
«когда понадобится подрядчик, буду рад помочь»).
* Не повторять одну и ту же синтаксическую конструкцию («без единого X» /
«без единого Y») в соседних предложениях одного письма.
* Если зацепка — законченная клауза со своим подлежащим (например, «клиенты
отмечают, что...») — after «Зацепило» ставится «то, что»; если начинается с
«как» — просто запятая. Если это голая именная фраза без глагола («у вас
международный сертификат...») — нужно «у вас» перед ней, иначе грамматически
не клеится.
* Единый стиль: короткое тире «–» вместо длинного «—» **везде**, включая
внутренние документы, не только в письмах.

## 4\. Отправка — `send\_campaign.py`

* Только стандартная библиотека Python (`smtplib`, `csv`, `email`, `fcntl`) — без
`pip install`. На хостинге Python 3.8.8, алиас `python3` **не виден в
cron** — нужен полный путь `/opt/python/python-3.8.8/bin/python`.
* Пароль — только через переменные окружения (`SMTP\_USER`, `SMTP\_PASSWORD`),
никогда в коде. Локальный шаблон для Windows — `set\_env.ps1` — реальный
пароль вписывается только в свою копию, не выкладывать.
* **Файловая блокировка** (`fcntl.flock`, эксклюзивная, на файл
`send\_campaign.lock`) — если предыдущий запуск ещё не завершился, новый тихо
выходит, не трогая CSV. Добавлена после того, как параллельные ручные тесты
привели к дублю письма (см. «Реальные результаты» выше).
* **Прогрев домена**: 5→5→8→8→12→12→15→18→20→20→25→25→30 писем/день, дальше
по 30. День прогрева считается от даты первого реально отправленного
письма (по `sent\_date` в самом CSV) — можно останавливать и продолжать.
* Пауза 3-8 минут между письмами, только рабочие часы буднего дня (10-18,
пн-пт) — проверяется и в начале запуска, и **повторно внутри цикла**
отправки (на случай, если долгий прогон перевалит за 18:00).
* `--dry-run` / `--limit N` / `--ignore-window` — для тестов.

**Инфраструктура (настроено, подтверждено рабочим на реальной отправке):**

* Домен для рассылки — `stivinteriors.ru` (роль двух доменов и их нестыковка
разрешены — см. раздел «Что это» в начале документа). SMTP: `mail.hosting.reg.ru`, порт 465
(SSL). Ящик `alex@stivinteriors.ru` — тот же адрес, что указан в общих
контактах сайта.
* SPF/DKIM/DMARC подтверждены через mail-tester.com — **10/10**, DKIM PASS.
DMARC сейчас `p=none` (осознанно, на период прогрева) — ужесточать до
`quarantine`/`reject` только после нескольких недель без проблем.
* Домашний интернет **блокирует порты 465/587** (типично для российских
провайдеров) — рассылка идёт через SSH на хостинге, не с домашнего компьютера.
* Рабочая директория на сервере: `/var/www/u2240000/data/flatica` (специально
вне `data/www/`, чтобы CSV с email-адресами дизайнеров не раздавался через
веб-сервер).

**Cron — финальная рабочая настройка**, один раз в день:
```
5 10 * * 1-5    /var/www/u2240000/data/flatica/run_campaign.sh >> /var/www/u2240000/data/flatica/cron.log 2>&1
```
Скрипт сам решает изнутри, сколько писем отправить и с какими паузами —
плодить более частые cron-задания не нужно и рискованно (см. следующий пункт).

**Обёртка `run\_campaign.sh`** (обязательна, не вызывать `send\_campaign.py`
из cron напрямую):
```bash
#!/bin/sh
cd /var/www/u2240000/data/flatica || exit 1
. ./env.sh
exec /opt/python/python-3.8.8/bin/python send_campaign.py campaign2.csv
```

Имя CSV в последней строке — текущая рабочая пачка; при переходе к следующей
пачке оно правится прямо в `run\_campaign.sh` (на 11.07.2026 — `campaign2.csv`).

**Найденные и решённые проблемы cron** (пригодится, если что-то похожее
всплывёт снова):
* `source env.sh` не работает под cron — там часто `/bin/sh` (dash), а не
bash, и `source` там не существует. Нужен POSIX-вариант `. ./env.sh`.
* `cd ~/flatica` под cron ненадёжен — `~` не всегда разворачивается в
урезанном cron-окружении. Использовать только абсолютные пути.
* Редирект `>> log 2>&1` в конце цепочки команд захватывает только
**последнюю** команду — если что-то раньше по цепочке падает (например,
`cd`), в лог не попадёт вообще ничего. Решение — редирект на **внешнем**
вызове обёртки (`run\_campaign.sh >> cron.log`), а не внутри неё.
* Если после верно настроенного cron лог всё равно пуст — возможно, реальный
cron-демон на этом shared-хостинге игнорирует пользовательский `crontab` и
требует создавать задачу через раздел панели «Cron-задания» (актуально для
части конфигураций ISPmanager). У этого хостинга в итоге сработал обычный
`crontab -e`.

* Сообщения короткие (2-3 предложения), без темы, без официального
приветствия, без подписи в конце.

**Поиск профилей:** хэштеги (#дизайнинтерьера и похожие), гео-метки
конкретных ЖК, раздел «Похожие профили» в рекомендациях Instagram, отметки у
крупных отраслевых аккаунтов (AD Russia, Houzz Russia).

**Три варианта сообщения, чередуются по порядку** (1-й дизайнер за день —
А, 2-й — Б, 3-й — В, 4-й — снова А):
* **А** (прямой, с деталью): «Привет! Видел твой проект \[деталь], очень
круто сделано. Мы в «Стив Интерьеры» делаем мебель под дизайн-проекты,
работаем с дизайнерами по Москве. Бывает нужен ещё один надёжный подрядчик
по мебели?»
* **Б** (комплимент, мягкий заход): «Привет! \[Деталь проекта] это прямо
огонь. Я мебельщик («Стив Интерьеры»), делаем корпусную мебель под чужие
дизайн-проекты. Когда понадобится подрядчик по мебели, буду рад помочь.»
* **В** (через вопрос, обычно работает лучше всех): «Привет! А кто вам делал
мебель на \[деталь проекта]? Спрашиваю не просто так: мы сами делаем мебель
под дизайн-проекты.»

**Учёт** — таблица в самом `.docx` (дата, ник, ссылка, деталь, вариант,
ответил, заметки), заполняется вручную, отдельно от CSV-пайплайна email-рассылки.

# Реактивация старых клиентов – третий канал (мессенджер-кампания)

Кампания по старой клиентской базе (клиенты, которым ставили мебель 1–3 года
назад) через мессенджер, где велась переписка по проекту. Ведёт Александр,
с личного/рабочего номера, вручную.

## Цели (в порядке приоритета)

1. **Отзывы на Яндекс.Картах** – главная витрина при отсутствии шоурума.
   Прямая ссылка на вкладку отзывов (вставлена во все шаблоны):
   `https://yandex.ru/maps/org/stiv_interyery/152572712181/reviews/`
   (проверено 11.07.2026: `YANDEX_MAPS_URL` в `config.php` ведёт на карточку
   организации, без `/reviews/` – это два намеренно разных URL: на сайте
   карточка, в шаблонах кампании – прямая ссылка на вкладку отзывов).
2. **Рекомендательная программа** – довольные клиенты как источник тёплых лидов.
3. **Повторные продажи** – вторая единица мебели (после кухни – гардеробная и т.п.).
4. **Скрытая цель: выявление претензий.** Недовольный клиент, о котором мы не
   знаем, молча отговаривает знакомых; клиент, чью проблему сами нашли и решили,
   становится лояльнее того, у кого проблем не было (service recovery paradox).
   Претензии – не риск кампании, а её ценный результат.

## Файлы

Хранятся в папке `assets` (там же, где исходник логотипа `assets/logo.ai`).

* **`Стратегия_работы_со_старыми_клиентами.docx`** – полный документ (10 стр.):
  цели, принципы, сегментация, каскад со всеми ветками, 13 готовых шаблонов
  сообщений, сервисный протокол, механика рекомендательной программы, метрики,
  чек-лист перед запуском, шпаргалка-таблица всех сообщений (кому / условие /
  через сколько дней / текст) в конце.
* **`build_strategy_docx.js`** – генератор документа (Node.js, библиотека
  `docx` – тот же подход, что `kp_build.js`). Все правки – в скрипте, затем
  `node build_strategy_docx.js`.
* Есть параллельная markdown-версия того же содержания.

## Ключевые решения (не менять без причины)

* **Рамка первого касания – «бесплатный сервисный визит», а не просьба.**
  Мебель 2–3 лет требует регулировки (петли, доводчики, фасады) – предложение
  проверки даёт ценность, естественно вскрывает претензии и создаёт повод для
  последующей просьбы об отзыве.
* **Одно сообщение – одна мысль.** Никогда не объединять чек-ин + отзыв +
  рекомендации в одном сообщении: читается как «нам от вас что-то нужно»,
  а при наличии претензии просьба об отзыве приглашает негатив на Карты.
* **Отзыв просим только у явно довольных** (сказал «всё отлично» или после
  успешно закрытого сервисного визита). **Никакого вознаграждения за отзыв** –
  Яндекс режет рейтинг и удаляет покупные отзывы, а «отзыв за бонус» в
  переписке – компромат. Вознаграждение только за рекомендацию, приведшую к заказу.
* **Механика «перенесите ваши тёплые слова в отзыв»** – если клиент сам
  похвалил в переписке, просим перенести эти же фразы в отзыв (человеку не
  нужно ничего сочинять, текст уже написан). Применять постоянно при любой
  спонтанной похвале (после сдачи проекта и т.д.), не только в кампании.
  Старый «слащавый» шаблон благодарности (эмодзи, «Вы – лучшая!») заменён
  прямой версией без эмодзи и длинного тире.
* **Претензия ставит все просьбы на паузу.** В первом ответе на претензию
  запрещены «это не гарантийный случай», «гарантия закончилась», оправдания
  и условия – сначала смотрим, потом обсуждаем. Мелочь чиним бесплатно даже
  вне гарантии; серьёзный дефект по нашей вине – бесплатно; эксплуатационный –
  ремонт по себестоимости/со скидкой.
* **Пауза между просьбами обязательна** (отзыв → 3–7 дней → рекомендации);
  после двух «тишин» на первое касание клиент закрывается навсегда.
* **Темп: 5–10 сообщений в день**, будни 11:00–19:00, с обычного номера
  (массовые однотипные сообщения в WhatsApp чреваты блокировкой). Старт –
  тестовая партия из 3–5 самых лояльных клиентов сегмента А.
* Тон: без длинного тире (глобальное правило проекта), без эмодзи,
  персонализация обязательна (имя, предмет, адрес/ЖК, год установки).

## Сегментация

|Сегмент|Кто|Заход|
|-|-|-|
|А (основной)|2–3 года после установки|бесплатная сервисная проверка|
|Б|около года|чек-ин «как мебель в жизни» (сервисная рамка натянута – мебель новая)|

## Каскад (кратко)

Касание 1 (сервис/чек-ин) → по реакции: «всё отлично» → касание 2 (отзыв) →
касание 3 (рекомендательная программа); «есть проблема» → сервисный протокол →
после решения обратно в каскад; тишина → одно напоминание → закрыт.
Полная схема, все ветки и тексты – в docx.

# Реальная схема сделки (замер → проект → договор) — зафиксировано 21.07.2026

Как на самом деле устроено доведение клиента до заказа. Раньше в документе этого
не было; в тот же день сайт и скрипт общения приведены в соответствие с этой схемой.

* **Замер бесплатный и ни к чему не обязывает.** На замере Станислав снимает точные
  размеры, рисует эскиз от руки, обсуждает состав и решения и называет ориентир по
  цене. 3D-визуализацию (PlanPlace и т.п.) вживую не делаем: это 20–30 минут паузы,
  убивающей динамику встречи. Визуализация готовится позже, на оплаченном этапе проекта.
* **99% сделок идут через предварительный договор — 25 000 ₽.** Обязательство по нему:
  полная документация проекта (спецификация материалов + 3D-визуализация). При заказе
  мебели эти 25 000 ₽ засчитываются в первый взнос, то есть «проект бесплатный при заказе»,
  но не «без денег». Формулировка для клиента: деньги входят в стоимость мебели.
* **Редкий путь — простой понятный проект:** на месте основной договор + 65% предоплата,
  затем Приложение 1 (спецификация) подписывается дистанционно или на втором визите.
* **Цена до готовой спецификации — ориентир, а не финал.** Финальная цена фиксируется в
  договоре после согласования проекта и дальше меняется только при изменении проекта по
  просьбе клиента. У ориентира есть срок годности: после паузы пересчитываем перед стартом
  и говорим об этом клиенту.
* **Правило коммуникации.** Предварительный договор и цифры (25 000 ₽, 65%) — разговор на
  замере глаза в глаза, на сайт их не выносим. Речь для замера — в «Скрипт общения с
  клиентами.md», раздел «Перевод в проект».
* **Урок (почему схема именно такая).** Два договора потеряны (деньги возвращены) из-за
  обещания неизменной цены до проектирования и из-за устаревшего ориентира после двух
  месяцев паузы. Теперь цена всегда подаётся как ориентир + актуализация, а неизменной
  называется только цена, зафиксированная в договоре после согласования проекта.
* **Правки на сайте под эту схему (21.07.2026).** Убраны обещания, которых нет:
  «3D-проект бесплатно» / «Замер и 3D-проект — бесплатно» (about.php, index.php, land-mos),
  «3D-модель на месте» (about.php, заменено на «Обсуждение и эскиз на замере»), «цена не
  меняется ни после замера, ни в процессе» (land-ng, land-cc, заменено на фиксацию в
  договоре после согласования проекта). Слово «бесплатно» у проекта убрано везде; фраза
  «стоимость проекта входит в стоимость мебели» на сайте не используется (не продаёт).
  Разбивка оплат 65/25/10 в таймлайне about.php сохранена по решению Александра. Полный
  список правок — «План_правок_под_реальную_схему.md».
