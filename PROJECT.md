## Как ведётся этот документ

Как только в проекте происходит значимое изменение (новая архитектура, новый
файл/скрипт, изменение конвенции или принципа, важное решение с обоснованием),
которое может пригодиться позже для понимания проекта, — **это нужно сначала
предложить записать сюда**, и только после согласия действительно вносить
правку. Не молча и не «на автомате» — документ должен отражать то, что
реально согласовано, а не всё подряд, что происходило в сессии.

## GitHub

Код сайта (`/var/www/u2240000/data/www/stivmebel.ru`) — git-репозиторий,
подключённый к GitHub:

* **Репозиторий:** `git@github.com:mrdronru/stivmebel.git`
* **Ветка:** `main`

**Сохранение изменений — один скрипт.** На сервере, на уровень выше `www`
(там же, где сам `stivmebel.ru`), лежит исполняемый файл `stivm` — быстрый
коммит + пуш одной командой:

```bash
./stivm
```

(если `Permission denied` — один раз `chmod +x stivm`, дальше работает.)

Скрипт сам переходит в `SITE_DIR`, делает `git add .`, коммитит с
автосообщением вида `Изменение от ДД.ММ.ГГГГ ЧЧ:ММ` и пушит в `origin/main`.
Если изменений нет — просто сообщает об этом и ничего не коммитит.

Посмотреть историю изменений:
```bash
cd /var/www/u2240000/data/www/stivmebel.ru
git log --oneline -10
```

**Почему это вообще завели:** однажды рекомендация Claude в одной из сессий
привела к стиранию актуальных файлов сайта, а восстановить было неоткуда —
в памяти/контексте той сессии их тоже уже не было. GitHub здесь не «для
галочки», а реальная страховка на именно такой случай. Из этого — практическое
правило на будущее: **перед любыми потенциально разрушительными действиями
(перезапись, массовое удаление, рефакторинг структуры файлов) сначала
`./stivm`**, чтобы был откат, и лишь затем вносить изменения.

## Стек

* **PHP** — шаблонизация через `include 'header.php'` / `include 'footer.php'`
* **Vanilla JS** — без фреймворков
* **CSS custom properties** — единая дизайн-система
* Хостинг: **Reg.ru** shared. Для форм сайта (`send.php`): `fsockopen` к внешним SMTP заблокирован, `curl` к внешним хостам работает, `mail()` принимает письма, но они не доходят (нет DKIM на уровне PHP-отправки) — поэтому заявки с форм идут в Telegram Bot API, а не на почту. Это ограничение не относится к отдельному процессу массовой рассылки дизайнерам (Python `smtplib` через SSH — см. раздел «B2B-рассылка дизайнерам» в конце файла), там SMTP настроен и работает.

### config.php — единый источник важных данных

**Все чувствительные и часто меняемые настройки хранятся только в `config.php`.** Этот файл подключается через `require\_once \_\_DIR\_\_ . '/config.php'` в начале каждой страницы и в `send.php`.

Что хранится в config.php:

* `SITE\_PHONE`, `SITE\_PHONE\_HREF` — номер телефона (меняйте только здесь)
* `SITE\_EMAIL`, `SITE\_EMAIL\_HREF` — email
* `SOCIAL\_TG`, `SOCIAL\_VK`, `SOCIAL\_MAX` — ссылки на соцсети (иконки в футере и попап-форме)
* `SOCIAL\_VK\_PAGE`, `SOCIAL\_TG\_CHANNEL`, `YANDEX\_MAPS\_URL` — ссылки на страницы/каналы и карту (текстовые ссылки в футере и блоке «Контакты»)
* `SITE\_NAME`, `SITE\_TAGLINE` — название и слоган
* `YANDEX\_METRIKA\_ID` — ID счётчика Яндекс.Метрики
* `TG\_TOKEN`, `TG\_CHAT\_ID` — токен Telegram-бота и ID чата получателя заявок
* `PRICES` — цены «от» (₽/п.м.) по категориям: wardrobes (шкафы и шкафы-купе,
включая прихожие), kitchens (кухни), closets (гардеробные), kids
(детские/столы/стеллажи, `from => null` = «расчёт после замера»). Единый
источник для блока «Наши цены» на главной, таблицы ориентиров и meta
description на `price.php`. «Шкафы» на главной и «Шкафы и шкафы-купе» на
/price — одна категория `wardrobes` с одной ценой. Добавлено 12.07.2026;
тогда же категории приведены в порядок: бывшая строка «Прихожие, шкафы»
(от 44 900) влита в wardrobes, её цена стала ценой категории, прежняя цена
wardrobes 89 000 ₽ убрана с сайта полностью (решение Александра).
* `price\_from($key)` — функция форматирования: `price\_from('kitchens')` →
«от 59 500 ₽»; для `from => null` возвращает «расчёт после замера».

**Важно:** не хардкодить контакты, токены, ID или цены напрямую в страницах — всё через константы из config.php. Цены менять только в массиве `PRICES` — главная, страница цен и её description обновятся сами.

Медиафайлы:

* `images/` — фото для index (k01–k08.jpg, m01–m06.jpg)
* `gallery/` — фото галереи, формат `NNNN-category` (`0001-kitchen.jpg` и далее; нумерация растёт по мере добавления, на июль 2026 — до `0147-other.jpg`)
* `video/` — видео: `NNNN.mp4` + `NNNN.jpg` (обложка) + опционально `NNNN.md` (подпись).
Список читается автоматически через `glob()` в `video.php` — без `.md` видео
показывается без подписи. См. раздел «Видео» ниже.
* `hello.mp4` — видео Станислава: используется на `about.php` (в круге) и на `land-ng.php` (полноэкранное в блоке о Станиславе). На `land-cc.php` в этом блоке вместо видео — фото `stas01.png`

|Файл|Что содержит|

**Важно:** `gallerystyle.css` дублирует стили `.modal-field input/textarea` из `style.css` с `!important` — намеренно, иначе браузерные дефолты перебивают поля рамками.

Адаптив — основной брейкпоинт: `@media (max-width: 900px)`. Дополнительно в `gallerystyle.css` и `favoritesstyle.css` есть `@media (max-width: 500px)` — осознанное исключение для переключения сетки фото с 2 колонок на 1 на очень узких экранах, а не случайный третий брейкпоинт.

### Цепочки подключения и риск переопределений

Файл позже в цепочке побеждает при равной специфичности селектора — это
нормальный каскад CSS, но именно на стыке файлов рождаются «тихие» баги:
страничный файл переопределяет правило из общего `style.css`, и
переопределение либо теряет смысл исходного значения, либо расходится по
брейкпоинтам с третьим файлом в цепочке.

|Страница|Цепочка|
|-|-|
|index.php|style.css|
|about.php|style.css → aboutstyle.css|
|catalog.php|style.css → gallerystyle.css|
|favorites.php|style.css → gallerystyle.css → favoritesstyle.css|
|video.php|style.css → videostyle.css|
|price.php|style.css → pricestyle.css|
|abcd.php / abc/\*.html|style.css → abcstyle.css|
|land-ng.php|style.css → landing.css|
|land-cc.php|style.css → landing.css|

`favorites.php` — единственная страница с цепочкой длиной три файла, то есть
с наибольшим риском такого конфликта. Реальный случай: `favoritesstyle.css`
на мобильном переопределял `.fav-header { padding: 3rem ...}`, обнуляя
`padding-top: 7rem`, который `.gallery-header` в `gallerystyle.css` намеренно
ставил под высоту `position: fixed` навигации (`nav` в `style.css`) — в
результате шапка перекрывала заголовок страницы. Это и есть целевой класс
багов для процедуры проверки ниже (см. чеклист «CSS — консистентность»).

## Квиз (land-ng.php — только этот лендинг, на land-cc.php квиза нет)

Встроенный квиз-модал, открывается кнопкой «Рассчитать стоимость» в герое.
6 шагов: тип мебели → метраж → стиль → бюджет → срок → контакт.
На шаге 6 — выбор способа связи: Телефон, Telegram, ВКонтакте, Макс, Email.
Ответы собираются в `comment`, отправляются в `send.php` с `source='quiz'`.
В Telegram приходит с иконкой 🎯 и пометкой «Квиз».

## send.php — отправка заявок

**Telegram Bot API через curl.**

```
Bot token: см. BotFather
Chat ID:   см. BotFather (@mrdronru — Alexandr Dranitskiy, владелец)
```

Принимает JSON: `name`, `phone`, `type`, `comment`, `photos\[]`, `source`.
Возвращает `{"ok":true}` или `{"ok":false,"error":"..."}`.

Поле `source`: `main` (форма index), `popup` (попап), `gallery` (модал галереи), `quiz` (квиз на land-ng).

**`parse\_mode` не используется — сообщения отправляются как обычный текст.** Раньше использовался `Markdown` с неполным экранированием (`tg\_escape`), из-за чего спецсимволы в именах/телефонах/комментариях могли ломать запрос к Telegram API. Теперь `parse\_mode` убран из вызова API целиком, разметка `\*жирный\*` не используется — пользовательский ввод безопасен в любом виде.

## Видео

### Структура

```
video/           папка
├── NNNN.mp4     видеофайл (обязательно)
├── NNNN.jpg     обложка/постер (обязательно)
└── NNNN.md      подпись — title + описание (опционально)
```

Список видео в `video.php` читается автоматически через `glob('video/\*.mp4')`
при каждой загрузке страницы — добавлять файлы вручную в код не нужно.
Сортировка — по имени файла (`0001`, `0002`...), как и раньше.

Если рядом с `NNNN.mp4` нет `NNNN.md` — видео всё равно показывается в ленте,
просто без заголовка и описания (блок `.video-meta` не выводится).

### Формат NNNN.md

```
---
title: Название видео
---

Текст описания в одну-несколько строк.
```

Описание выводится как обычный текст без markdown-разметки (жирный/курсив/
ссылки сознательно не поддерживаются здесь — `video.php` пропускает результат
`parseMarkdown()` через `strip\_tags()`). Формат файла такой же простой, как у
заголовка — поддерживается только поле `title` в frontmatter.

**Статус:** подписи `0001.md`–`0020.md` подготовлены и должны лежать в `video/`
рядом с соответствующими `.mp4`/`.jpg`. При добавлении видео №21 и далее нужно
просто положить новый `NNNN.mp4` + `NNNN.jpg` (+ опционально `NNNN.md`) — список
на странице обновится сам.

### Генерация страниц

```bash
php build.php
```

Запускать после каждого добавления или изменения MD-файлов.
Скрипт пересоздаёт все HTML и обновляет `abc/index.json`.

Каждая страница термина генерируется через настоящие `header.php`/`footer.php`
(`include` + `ob\_start()`/`ob\_get\_clean()`), а не через отдельную копию шаблона —
любое изменение шапки/футера сайта подхватывается статьями Азбуки автоматически
при следующем запуске `build.php`, без необходимости синхронизировать вручную.
Используется `$basePath = '../'` (страницы лежат в подпапке `abc/`) и явный
`$canonical\_path` для `og:url`/`canonical` — при генерации из CLI
`$\_SERVER\['REQUEST\_URI']` не определён.

## Лендинг land-cc.php — архитектура и решения

Второй лендинг, для премиум-аудитории ЖК Сердце Столицы (бизнес-класс,
квартиры 21–50 млн ₽). Делит общий `landing.css` с `land-ng.php`, но другая
тональность, другая структура hero и нет квиза.

### Кейс проекта

Единственный пока выполненный заказ в этом ЖК — шкаф высотой 3 метра с
монолитными дверьми без антресоли (МДФ в эмали, интегрированная ручка на всю
ширину фасада, фурнитура Blum, даже крепёжные саморезы покрашены в цвет
фасада). Заказчица — Марина. Видеообзор: `video/0010.mp4` (42 сек, описание в
`video/0010.md`). Фото кейса: `gallery/0043–0046-wardrobe.jpg`.

### Шапка

Та же `.ln-nav`, что у `land-ng.php` — переиспользуется без изменений.
Кнопка — «Обсудить проект» (не «Бесплатный замер», как на land-ng).

### Структура секций

1. **Hero** — фоновое видео `video/0010.mp4` (автоплей, без звука, зациклено),
постер `gallery/0044-wardrobe.jpg`, заголовок, кнопки «Обсудить проект» +
«▶ Видео со звуком» (открывает попап с тем же видео, уже со звуком и
нативными controls — см. «Видео-попап» ниже), стрелка-подсказка скролла
внизу (`l-scroll-hint`, переиспользована из общих стилей)
2. **Case** (`.lcc-case`) — текст про шкаф 3 метра без антресоли, с цитатой в
конце (`.lcc-case-quote`, золотой левый бордер)
3. **Portfolio photos** (`.lcc-photo-grid`) — простая сетка 2×2, все 4 фото
одинакового размера (aspect-ratio 4/3), у каждого своя подпись
4. **Story** (`.lcc-story`) — тёмный блок (фон `var(--ink)`), цитата Марины
золотым курсивом (`.lcc-blockquote`), два абзаца текста
5. **About** — фото Станислава (`images/stas01.png`, прозрачный фон) вместо
видео, как на land-ng. Текст от первого лица
6. **UTP** — 4 преимущества (не 5, как на land-ng): расписание под клиента,
организация на себя, «возможно почти всё», цена в договоре
7. **CTA** — финальный блок, фон — то же фото `gallery/0044-wardrobe.jpg`

### Видео-попап (только land-cc)

Кнопка в hero «▶ Видео со звуком» открывает `#videoModal` — затемнённый
оверлей с тем же видео `video/0010.mp4`, но уже с `controls` и звуком.
JS-функции `openVideoModal()` / `closeVideoModal()`, закрытие по клику на фон
или `Escape`. Это решение взамен первоначальной идеи доигрывать звук прямо в
фоновом hero-видео — авто-переключение `muted` на фоновом видео визуально
ломало композицию (фон при этом ещё и должен оставаться `cover`/`loop`).

### Фото Станислава на этом лендинге

`images/stas01.png` — должен быть **с прозрачным фоном** (PNG), это
сознательное отличие от `land-ng`, где используется видео в рамке.
**История правки:** изначально пробовали CSS-маски (`mask-image` с
radial-gradient) чтобы низ фото плавно растворялся в фон блока — после
нескольких неудачных итераций (направление градиента упорно путалось,
результат выглядел противоположным замыслу) от этого подхода отказались.
**Все CSS-эффекты (маски, градиенты) с фото убраны.** Если нужно плавное
исчезновение низа — готовить эффект заранее в редакторе (Photopea/Figma) и
заливать уже готовый PNG, а не пытаться сделать через CSS-маску на лету.

### Тональность — отличие от land-ng

Здесь принципиально другой регистр: сдержанный, «архитектурное бюро», без
разговорных оборотов. Ключевая мысль из ТЗ на тексты: **не продавать
характеристики, а говорить о неизбежности правильного решения** — ориентир
на интонацию брендов вроде Boffi/Poliform/Minotti. Никогда не упоминать и не
принижать конкурентов — позиционирование строится только через свой кейс,
без сравнений.

Финальные согласованные тексты:

* **H1:** «Есть вещи, которые должны быть безупречными. Мы создаём мебель, в
которой всё — именно так, как должно быть.»
* **Case quote:** «Потому что роскошь — это не то, что видно сразу. Это
отсутствие вещей, которые могли бы раздражать потом.»
* **Story quote (Марина):** «Понравились работы и стиль. Сразу почувствовала,
что всё получится»
* **УТП:** «Возможно почти всё» (не «Никаких неосуществимых проектов» — более
мягкая, не категоричная формулировка)

### Нет на этом лендинге (отличия от land-ng)

* Нет квиза и кнопки «Рассчитать стоимость»
* Нет видео Станислава (`hello.mp4`) — только статичное фото
* Нет блока District (вместо него — Case с историей конкретного проекта)
* УТМ-метки и попап-форма — общие с остальным сайтом (`footer.php`), без
отдельной настройки

### Идея

PHP-страницы сейчас рендерятся заново при каждом обращении к сайту — это лишняя
нагрузка там, где контент не меняется от запроса к запросу. Предлагается по
аналогии с Азбукой (`build.php` уже генерирует `abc/\*.html` из `.md` через
`header.php`/`footer.php`) расширить этот подход на весь сайт: превратить
контентные страницы в статический HTML, оставив PHP только там, где он
объективно нужен — серверная логика и секреты.

### Что остаётся на PHP всегда

* `send.php` — обработка формы, отправка в Telegram (токен бота, server-side по определению).
* `config.php`, `build.php`, `markdown-helpers.php` — служебные файлы, не отдаются напрямую.
* Любая логика, которая обязана выполняться на сервере при каждом запросе.

### Что переводится в статику

`index.php`, `about.php`, `price.php`, `catalog.php`, `video.php`, `favorites.php`,
`abcd.php` (сам список терминов, не только сгенерированные статьи), `politika.php`,
`soglasie.php` — весь контентный слой сайта.

### Решённые вопросы (на этапе обсуждения)

* **Переход одним рефакторингом**, не постранично — весь сайт за один заход.
* **URL без `.html`** — красивые адреса (`/about`, `/gallery`) через `.htaccess`,
не показывать расширение пользователю.
* **Пересборка после правок — нормальная практика**, уже привычна по работе
с Азбукой: после любого изменения `header.php`/`footer.php`/другой общей части
нужно заново запускать `build.php`, чтобы изменения попали на все страницы.
Это осознанно принятый компромисс, не недостаток подхода.

### Технические задачи, которые предстоят

1. **Маршрутизация в `.htaccess`.** ✅ Выполнено досрочно 11.07.2026 в рамках
SEO-доработок (см. раздел «SEO-доработки» ниже): чистые URL теперь канонические.
Для перехода на статику останется только поменять приоритет в правилах: если

2. **Переписать все внутренние ссылки сайта.** ✅ Выполнено 11.07.2026: все

3. **301-редирект со старых `.php`-адресов.** ✅ Выполнено 11.07.2026 —
см. раздел «SEO-доработки». Проиндексированные `.php`-адреса и `abc/*.html`
отдают 301 на чистые URL, SEO-вес переносится.
4. **`catalog.php` и `video.php` — главное изменение в рабочем процессе.**
Сейчас обе страницы читают список файлов через `glob()` при каждом заходе —
это и есть вся их «логика». В статике это означает: добавили фото/видео на

«просто залил файл — он сразу на сайте». Владелец предупреждён и согласен

5. **`favorites.php`.** Список избранного хранится в `localStorage` браузера —
это уже клиентская логика, страница переносится в статику почти без проблем.
6. **`$canonical\_path` для каждой страницы.** Та же история, что уже решена для
Азбуки — при генерации из CLI `$\_SERVER\['REQUEST\_URI']` не определён, нужно
явно прописывать canonical/og:url для каждой генерируемой страницы.
7. **`build.php` должен научиться генерировать не только статьи Азбуки.**
Сейчас генератор заточен под `abc/\*.md → abc/\*.html`. Нужно расширить его
(или сделать параллельный генератор) для `index.html`, `about.html`,
`gallery.html` и т.д. — каждая страница сохраняет свою текущую бизнес-логику
(например, `glob()` для фото в галерее), просто результат уходит в файл,
а не в ответ на HTTP-запрос.

## SEO-доработки (11.07.2026)

Повод — отчёт стороннего SEO-чекера. Большинство его замечаний оказались
ложными (H1 присутствует на всех страницах, «встроенный CSS» — шум, короткий
H1 на страницах Азбуки — корректная семантика глоссария), но три реальных
пункта закрыты за один проход:

**1. Канонизация URL — чистые адреса стали основными.**

* `.htaccess`: 301-редиректы `/страница.php → /страница`, `/index.php → /`,
`/abc/термин.html → /abc/термин` и срезание хвостового слеша у не-директорий.
Редиректы построены на `THE_REQUEST` (не `REQUEST_URI`), чтобы срабатывать
только на прямые запросы браузера и не зацикливаться с внутренними рерайтами.
* **`send.php` исключён из редиректов** — его вызывают формы POST'ом через
`fetch('send.php')`, 301 сломал бы отправку заявок. Не убирать исключение.
* Все внутренние ссылки переведены на чистые URL (header, footer, страницы,
лендинги, 404, генератор статей). Ссылка «Контакты» строится как
`($basePath ?: '/') . '#contact'`.
* `build.php`: sitemap генерируется с чистыми URL, `$canonical_path` статей
Азбуки — без `.html`, ссылки prev/next и «назад» внутри статей — чистые.
**После любой заливки этих правок обязателен `php build.php`** — пересобрать
статьи и sitemap.
* Ссылка в Яндекс.Директе (`/land-ng`) уже была чистой — кампании не затронуты.
* **Конфликт имён страница/папка:** `/video` совпадает с папкой `video/`
(ролики), `/gallery` — с папкой `gallery/` (фото). Общий рерайт требует
«не директория», поэтому без явных правил Apache добавлял слеш и отдавал 403,
а первая попытка чинить это редиректом `/video/ → /video` дала петлю с
`mod_dir` (он сам 301-ит `/video → /video/`). Итоговое решение (петля
невозможна ни при каком поведении хостинга): в `.htaccess` — `DirectorySlash
Off` (если хостинг вернёт 500 на директиву, строку можно удалить, схема
переживёт) и `RewriteRule ^video/?$ video.php [L]` — страница отдаётся и по
`/video`, и по `/video/` без редиректа между ними; каноничность держит явный
`$canonical_path = 'video'` в `video.php`. **Никогда не добавлять 301 между
`/video` и `/video/`.** `/gallery` и `/gallery/` — 301 на `/catalog` (цель не
директория, петли нет). Прямые ссылки на файлы (`/video/0001.mp4`,
`/gallery/0002-kitchen.jpg`) не затрагиваются. **При добавлении новой страницы
не называть её именем существующей папки** (images, assets, testimonials, abc)
— либо повторять схему video: явный рерайт + явный `$canonical_path`.

**2. Расширены `<title>` внутренних страниц** (было 21–28 символов, стало

## 1\. Скрапер — `flatica\_designers.py`

Собирает профили дизайнеров со страницы списка (например,
`https://flatica.ru/professionals/speciality.dizayn-interyera\~city.moskva`).

**Ключевая архитектура:**

* Список дизайнеров требует настоящего браузера (Playwright) — сайт грузит
карточки через JS с задержкой 5-8 сек после докрутки до конца страницы.
Скрипт докручивает до текущего конца → ждёт с опросом высоты страницы
(`LOAD\_WAIT\_MAX=10 сек`, `LOAD\_POLL\_INTERVAL=1 сек`) → повторяет, пока
`STALL\_LIMIT=3` докрутки подряд не дают роста.
* **Страницы профиля НЕ требуют кликов вообще** — flatica.ru рендерит всё на
сервере (SSR): биография, первая страница проектов и отзывов уже есть в
обычном HTML. Разбор идёт через `BeautifulSoup` по реальным id-якорям
(`id="projects"`, `id="reviews"`, `id="about"`), без единого клика по
вкладкам. Это было НЕ очевидно и стоило \~15 итераций отладки.
* Телефон достаётся из ссылки на WhatsApp (`wa.me/...`) в HTML — в контактах
он всегда замаскирован звёздочками и требует клика, но в WhatsApp-ссылке
уже в открытом виде.
* Имя берётся из `<h2>`, не `<h1>` — `<h1>` на странице flatica — задвоенная
карточка в подвале («Дмитрий КрассаДмитрий Красса»).

### Правила составления «лести» (зацепки) — накопленный опыт ручной вычитки

После первого автопрохода по всем 140 строкам выяснилось: у Claude нет
доступа к фотографиям на странице профиля Флатики, только к тексту. Из-за
этого большая часть автоматически сгенерированных зацепок оказалась не
деталью проекта, а фактом о компании (стаж, регалии, география, философия) —
и такие факты плохо стыкуются со стандартной концовкой «Видно, что решение
продиктовано не только эстетикой, но и тем, как этим будут пользоваться»,
которая по смыслу требует именно детали решения, а не общего факта.

**Рабочий процесс, который сложился:** Александр сам смотрит фото проекта на
Флатике и присылает Claude либо готовую деталь, либо описание/фото — Claude
формулирует по нему короткую зацепку. Это надёжнее, чем просить Claude
угадывать деталь по одному тексту.

**Правила формулировок, выведенные через правки:**
* Назвать одну конкретную деталь и **не разжёвывать**, почему она хороша —
короче почти всегда лучше.
* Различать деталь проекта (визуальная, конкретная — подходит под стандартную
концовку) и факт о компании (стаж, награды, позиционирование — под неё не
подходит, нужна другая связка или вообще без неё).
* Вопросы — **без отрицания** («Не нужен ли...» провоцирует ответ «не нужен»;
переформулировать в утвердительную форму: «Бывает нужен...»).
* Без расплывчатых обещаний («если когда-нибудь будет нужен» → конкретнее:
«когда понадобится подрядчик, буду рад помочь»).
* Не повторять одну и ту же синтаксическую конструкцию («без единого X» /
«без единого Y») в соседних предложениях одного письма.
* Если зацепка — законченная клауза со своим подлежащим (например, «клиенты
отмечают, что...») — after «Зацепило» ставится «то, что»; если начинается с
«как» — просто запятая. Если это голая именная фраза без глагола («у вас
международный сертификат...») — нужно «у вас» перед ней, иначе грамматически
не клеится.
* Единый стиль: короткое тире «–» вместо длинного «—» **везде**, включая
внутренние документы, не только в письмах.

## 4\. Отправка — `send\_campaign.py`

* Только стандартная библиотека Python (`smtplib`, `csv`, `email`, `fcntl`) — без
`pip install`. На хостинге Python 3.8.8, алиас `python3` **не виден в
cron** — нужен полный путь `/opt/python/python-3.8.8/bin/python`.
* Пароль — только через переменные окружения (`SMTP\_USER`, `SMTP\_PASSWORD`),
никогда в коде. Локальный шаблон для Windows — `set\_env.ps1` — реальный
пароль вписывается только в свою копию, не выкладывать.
* **Файловая блокировка** (`fcntl.flock`, эксклюзивная, на файл
`send\_campaign.lock`) — если предыдущий запуск ещё не завершился, новый тихо
выходит, не трогая CSV. Добавлена после того, как параллельные ручные тесты
привели к дублю письма (см. «Реальные результаты» выше).
* **Прогрев домена**: 5→5→8→8→12→12→15→18→20→20→25→25→30 писем/день, дальше
по 30. День прогрева считается от даты первого реально отправленного
письма (по `sent\_date` в самом CSV) — можно останавливать и продолжать.
* Пауза 3-8 минут между письмами, только рабочие часы буднего дня (10-18,
пн-пт) — проверяется и в начале запуска, и **повторно внутри цикла**
отправки (на случай, если долгий прогон перевалит за 18:00).
* `--dry-run` / `--limit N` / `--ignore-window` — для тестов.

**Инфраструктура (настроено, подтверждено рабочим на реальной отправке):**

* Домен для рассылки — `stivinteriors.ru` (роль двух доменов и их нестыковка
разрешены — см. раздел «Что это» в начале документа). SMTP: `mail.hosting.reg.ru`, порт 465
(SSL). Ящик `alex@stivinteriors.ru` — тот же адрес, что указан в общих
контактах сайта.
* SPF/DKIM/DMARC подтверждены через mail-tester.com — **10/10**, DKIM PASS.
DMARC сейчас `p=none` (осознанно, на период прогрева) — ужесточать до
`quarantine`/`reject` только после нескольких недель без проблем.
* Домашний интернет **блокирует порты 465/587** (типично для российских
провайдеров) — рассылка идёт через SSH на хостинге, не с домашнего компьютера.
* Рабочая директория на сервере: `/var/www/u2240000/data/flatica` (специально
вне `data/www/`, чтобы CSV с email-адресами дизайнеров не раздавался через
веб-сервер).

**Cron — финальная рабочая настройка**, один раз в день:
```
5 10 * * 1-5    /var/www/u2240000/data/flatica/run_campaign.sh >> /var/www/u2240000/data/flatica/cron.log 2>&1
```
Скрипт сам решает изнутри, сколько писем отправить и с какими паузами —
плодить более частые cron-задания не нужно и рискованно (см. следующий пункт).

**Обёртка `run\_campaign.sh`** (обязательна, не вызывать `send\_campaign.py`
из cron напрямую):
```bash
#!/bin/sh
cd /var/www/u2240000/data/flatica || exit 1
. ./env.sh
exec /opt/python/python-3.8.8/bin/python send_campaign.py campaign2.csv
```

Имя CSV в последней строке — текущая рабочая пачка; при переходе к следующей
пачке оно правится прямо в `run\_campaign.sh` (на 11.07.2026 — `campaign2.csv`).

**Найденные и решённые проблемы cron** (пригодится, если что-то похожее
всплывёт снова):
* `source env.sh` не работает под cron — там часто `/bin/sh` (dash), а не
bash, и `source` там не существует. Нужен POSIX-вариант `. ./env.sh`.
* `cd ~/flatica` под cron ненадёжен — `~` не всегда разворачивается в
урезанном cron-окружении. Использовать только абсолютные пути.
* Редирект `>> log 2>&1` в конце цепочки команд захватывает только
**последнюю** команду — если что-то раньше по цепочке падает (например,
`cd`), в лог не попадёт вообще ничего. Решение — редирект на **внешнем**
вызове обёртки (`run\_campaign.sh >> cron.log`), а не внутри неё.
* Если после верно настроенного cron лог всё равно пуст — возможно, реальный
cron-демон на этом shared-хостинге игнорирует пользовательский `crontab` и
требует создавать задачу через раздел панели «Cron-задания» (актуально для
части конфигураций ISPmanager). У этого хостинга в итоге сработал обычный
`crontab -e`.

# Instagram Direct — второй канал рассылки дизайнерам

Параллельно с email-рассылкой на Флатике — поиск дизайнеров в Instagram и
личные сообщения в директ. Ведёт **Станислав**, не Александр. Полный гайд —
отдельный файл `Instagram\_dlya\_Stanislava.docx`.

**Ключевые отличия от email-канала:**
* Только вручную — никаких ботов/автоматизации (риск бана аккаунта намного
жёстче, чем спам-фильтры у почты).
* Темп — **5 сообщений в день** (реалистичный ориентир, следующий шаг вниз до
2-3/день на первые 1-2 недели — тот же принцип прогрева, что и у почтового
домена, только для самого аккаунта).
* Сообщения короткие (2-3 предложения), без темы, без официального
приветствия, без подписи в конце.

**Поиск профилей:** хэштеги (#дизайнинтерьера и похожие), гео-метки
конкретных ЖК, раздел «Похожие профили» в рекомендациях Instagram, отметки у
крупных отраслевых аккаунтов (AD Russia, Houzz Russia).

**Три варианта сообщения, чередуются по порядку** (1-й дизайнер за день —
А, 2-й — Б, 3-й — В, 4-й — снова А):
* **А** (прямой, с деталью): «Привет! Видел твой проект \[деталь], очень
круто сделано. Мы в «Стив Интерьеры» делаем мебель под дизайн-проекты,
работаем с дизайнерами по Москве. Бывает нужен ещё один надёжный подрядчик
по мебели?»
* **Б** (комплимент, мягкий заход): «Привет! \[Деталь проекта] это прямо
огонь. Я мебельщик («Стив Интерьеры»), делаем корпусную мебель под чужие
дизайн-проекты. Когда понадобится подрядчик по мебели, буду рад помочь.»
* **В** (через вопрос, обычно работает лучше всех): «Привет! А кто вам делал
мебель на \[деталь проекта]? Спрашиваю не просто так: мы сами делаем мебель
под дизайн-проекты.»

**Учёт** — таблица в самом `.docx` (дата, ник, ссылка, деталь, вариант,
ответил, заметки), заполняется вручную, отдельно от CSV-пайплайна email-рассылки.

# Реактивация старых клиентов – третий канал (мессенджер-кампания)

Кампания по старой клиентской базе (клиенты, которым ставили мебель 1–3 года
назад) через мессенджер, где велась переписка по проекту. Ведёт Александр,
с личного/рабочего номера, вручную.

## Цели (в порядке приоритета)

1. **Отзывы на Яндекс.Картах** – главная витрина при отсутствии шоурума.
   Прямая ссылка на вкладку отзывов (вставлена во все шаблоны):
   `https://yandex.ru/maps/org/stiv_interyery/152572712181/reviews/`
   (проверено 11.07.2026: `YANDEX_MAPS_URL` в `config.php` ведёт на карточку
   организации, без `/reviews/` – это два намеренно разных URL: на сайте
   карточка, в шаблонах кампании – прямая ссылка на вкладку отзывов).
2. **Рекомендательная программа** – довольные клиенты как источник тёплых лидов.
3. **Повторные продажи** – вторая единица мебели (после кухни – гардеробная и т.п.).
4. **Скрытая цель: выявление претензий.** Недовольный клиент, о котором мы не
   знаем, молча отговаривает знакомых; клиент, чью проблему сами нашли и решили,
   становится лояльнее того, у кого проблем не было (service recovery paradox).
   Претензии – не риск кампании, а её ценный результат.

## Файлы

Хранятся в папке `assets` (там же, где исходник логотипа `assets/logo.ai`).

* **`Стратегия_работы_со_старыми_клиентами.docx`** – полный документ (10 стр.):
  цели, принципы, сегментация, каскад со всеми ветками, 13 готовых шаблонов
  сообщений, сервисный протокол, механика рекомендательной программы, метрики,
  чек-лист перед запуском, шпаргалка-таблица всех сообщений (кому / условие /
  через сколько дней / текст) в конце.
* **`build_strategy_docx.js`** – генератор документа (Node.js, библиотека
  `docx` – тот же подход, что `kp_build.js`). Все правки – в скрипте, затем
  `node build_strategy_docx.js`.
* Есть параллельная markdown-версия того же содержания.

## Ключевые решения (не менять без причины)

* **Рамка первого касания – «бесплатный сервисный визит», а не просьба.**
  Мебель 2–3 лет требует регулировки (петли, доводчики, фасады) – предложение
  проверки даёт ценность, естественно вскрывает претензии и создаёт повод для
  последующей просьбы об отзыве.
* **Одно сообщение – одна мысль.** Никогда не объединять чек-ин + отзыв +
  рекомендации в одном сообщении: читается как «нам от вас что-то нужно»,
  а при наличии претензии просьба об отзыве приглашает негатив на Карты.
* **Отзыв просим только у явно довольных** (сказал «всё отлично» или после
  успешно закрытого сервисного визита). **Никакого вознаграждения за отзыв** –
  Яндекс режет рейтинг и удаляет покупные отзывы, а «отзыв за бонус» в
  переписке – компромат. Вознаграждение только за рекомендацию, приведшую к заказу.
* **Механика «перенесите ваши тёплые слова в отзыв»** – если клиент сам
  похвалил в переписке, просим перенести эти же фразы в отзыв (человеку не
  нужно ничего сочинять, текст уже написан). Применять постоянно при любой
  спонтанной похвале (после сдачи проекта и т.д.), не только в кампании.
  Старый «слащавый» шаблон благодарности (эмодзи, «Вы – лучшая!») заменён
  прямой версией без эмодзи и длинного тире.
* **Претензия ставит все просьбы на паузу.** В первом ответе на претензию
  запрещены «это не гарантийный случай», «гарантия закончилась», оправдания
  и условия – сначала смотрим, потом обсуждаем. Мелочь чиним бесплатно даже
  вне гарантии; серьёзный дефект по нашей вине – бесплатно; эксплуатационный –
  ремонт по себестоимости/со скидкой.
* **Пауза между просьбами обязательна** (отзыв → 3–7 дней → рекомендации);
  после двух «тишин» на первое касание клиент закрывается навсегда.
* **Темп: 5–10 сообщений в день**, будни 11:00–19:00, с обычного номера
  (массовые однотипные сообщения в WhatsApp чреваты блокировкой). Старт –
  тестовая партия из 3–5 самых лояльных клиентов сегмента А.
* Тон: без длинного тире (глобальное правило проекта), без эмодзи,
  персонализация обязательна (имя, предмет, адрес/ЖК, год установки).

## Сегментация

|Сегмент|Кто|Заход|
|-|-|-|
|А (основной)|2–3 года после установки|бесплатная сервисная проверка|
|Б|около года|чек-ин «как мебель в жизни» (сервисная рамка натянута – мебель новая)|
|В|проблемные в прошлом|решение вручную по каждому: не трогать, либо только сервис без последующих просьб|

## Каскад (кратко)

Касание 1 (сервис/чек-ин) → по реакции: «всё отлично» → касание 2 (отзыв) →
касание 3 (рекомендательная программа); «есть проблема» → сервисный протокол →
после решения обратно в каскад; тишина → одно напоминание → закрыт.
Полная схема, все ветки и тексты – в docx.
